# ThreadManage 实现方案（V1.1.0）

> 目标平台：IMX6ULL (ARM Linux) · 语言：C99 · 依赖：`pthread` + `ThreadQueue` + `RedBlackTree` + `KernelLinkedList`
>
> 本文档为**内部实现说明**，供二次开发、调试、Bug 定位使用。用户 API 层面请看 `readme.md` 与 `include/ThreadManage.h`；已知设计权衡与审查发现见 `AI审查结果.md`。
>
> **重点**：本文档以线程池为主线（占 90%），单线程创建部分（`ThreadAPI_ThreadCreate` 等）仅做简述。

---

## 一、总览

ThreadManage 分两块：

1. **单线程创建（`ThreadManage_Main.c`）**：对 `pthread_create` 的包装，围绕 `T_ThreadCreateConfig` 提供三档渐进属性配置（`eSetAttr` 0/1/2）与参数越界自动修正。~800 行，逻辑线性、无状态、无共享内存。
2. **线程池（`ThreadManage_Pool.c`）**：本文档主体。~1500 行，具备完整生命周期管理、动态扩缩容、多种任务提交语义、UAF 防护、运行统计。

线程池的核心设计目标：

- **零轮询**：worker 睡在 `ThreadQueueAPI_GetMsg` 的 timedwait 上（100ms tick 只用来检查空闲缩容），无 busy loop
- **单锁 + 单 cond**：一把 `tMutex` 保护所有共享状态，一个 `tCond` 广播所有状态变化，避免锁嵌套
- **原地扩缩**：不预分配 `iMaxNum` 个 worker 槽；按需 malloc `T_WorkerNode`、按需释放
- **计时与执行分离**：`ThreadQueue` 是通用环形队列，worker 消费；两者独立
- **安全销毁**：`iActiveCallers + iLiveThreadNum` 双计数，Destroy 等待双零后再释放同步原语

## 二、数据结构

### 2.1 关系图

```
┌───────────────────────────────────────────────────────────────────────┐
│  T_ThreadPoolHandle  (Main.h)                                         │
│                                                                       │
│   uiMagic = 0x5448504C ("THPL")   ← 防重复销毁 / 野指针                │
│   tConfig       : iMinNum/iMaxNum/iQueueMaxSize/iIdleTimeoutMs        │
│                                                                       │
│   tMutex        : 保护以下所有可变字段                                 │
│   tCond         : CLOCK_MONOTONIC，broadcast 唤醒 workers/waiters      │
│                                                                       │
│   iShutdown     : 0=运行 1=销毁中                                      │
│   iLiveThreadNum: 存活 worker 计数（含 exiting 但未 return 的）        │
│   iBusyThreadNum: 正在执行 task 的 worker 计数                         │
│   iTaskQueueLen : 镜像队列长度（与 tCond 判断条件同一锁保护）          │
│   iExpanding    : 单次扩容节流标志                                     │
│   iActiveCallers: AddTask* 系列在飞调用者计数（UAF 防护）              │
│   uiWorkerSeq   : worker 名称自增序号                                  │
│   tStats        : 峰值/累计快照                                        │
│                                                                       │
│   tWorkerTree   ──►  rb_root (RedBlackTree)                            │
│                       │                                                │
│                       ├─► T_WorkerNode #1 { tRbNode, tBusyEntry,      │
│                       │                    tThreadConfig, ptPool,     │
│                       │                    iIsBusy, iExiting,         │
│                       │                    iRegistered }              │
│                       ├─► T_WorkerNode #2                              │
│                       └─► ...                                          │
│                                                                       │
│   tBusyList     ──►  list_head (KernelLinkedList)                     │
│                       │                                                │
│                       ├─► T_WorkerNode #1.tBusyEntry                  │
│                       └─► T_WorkerNode #3.tBusyEntry                  │
│                                                                       │
│   ptTaskQueue   ──►  T_ThreadQueue  (ThreadQueue 库)                   │
│                       [ T_PoolTask*, T_PoolTask*, ... ] ← 环形         │
└───────────────────────────────────────────────────────────────────────┘
```

**关键点**：
- `T_WorkerNode` 同时属于**红黑树**（按 `pthread_t` 索引，O(log n) 查找）和**忙碌链表**（`tBusyEntry`，O(1) 插入/删除）。这两个数据结构都是侵入式的，节点内嵌在 `T_WorkerNode`，不额外 malloc。
- `ptTaskQueue` 里存的是 `T_PoolTask*` 指针，`T_PoolTask` 由 AddTask 分配、worker 消费后 free。
- `tCond` 是共享的：workers 在队列空时睡它（间接：`ThreadQueueAPI_GetMsg` 有自己的 cond；池 tCond 用于 `WaitAllDone`/`Destroy`/`CreateWorker` 的 `iRegistered` 同步/`AddTask` 的完成通知）。

### 2.3 T_WorkerNode 字段语义

| 字段 | 类型 | 语义 |
|------|------|------|
| `tThreadConfig` | `T_ThreadCreateConfig` | 传给 `ThreadAPI_ThreadCreate`；`tThreadPid` 由 pthread_create 回填 |
| `tRbNode` | `struct rb_node` | 红黑树节点，嵌入结构体 |
| `tBusyEntry` | `struct list_head` | 忙碌链表节点，嵌入结构体 |
| `ptPool` | `T_ThreadPoolHandle*` | 反向指针，worker 拿池句柄用 |
| `iIsBusy` | `int` | 1 = 正在执行 task，同时在 `tBusyList` 中 |
| `iExiting` | `int` | 1 = 已被标记退出（Resize 缩容 / 空闲超时） |
| `iRegistered` | `int` | 1 = 已插入 `tWorkerTree`，worker 主循环可进入 |

**并发约定**：`iIsBusy` / `iExiting` / `iRegistered` 都在 `ptPool->tMutex` 保护下改写。

由 AddTask 分配、入 `ptTaskQueue`，worker 出队执行、执行完 free。

## 三、同步原语与并发约定

### 3.1 单锁 + 单 cond

- 唯一互斥锁：`tMutex`
- 唯一条件变量：`tCond`（绑 `CLOCK_MONOTONIC`）
- ThreadQueue 内部另有一把锁 + 一个 cond，但对池层不可见；池 API 调用 `ThreadQueueAPI_PutMsg/GetMsg` 时**必须先释放 `tMutex`**，避免锁反转

### 3.2 锁保护范围

**所有** `T_ThreadPoolHandle` 的可变字段（除 `uiMagic` 的初值 / Destroy 时的清零外）都在 `tMutex` 内读写。

**`T_WorkerNode`** 的三个状态字段（`iIsBusy` / `iExiting` / `iRegistered`）也在 `tMutex` 内读写；配置字段（`tThreadConfig`，`ptPool`）在创建后不可变，无需锁。

### 3.3 广播时机

任何"可能影响其他等待方判断条件"的状态变化后 broadcast：

| 状态变化 | 谁在等 | 广播位置 |
|----------|--------|----------|
| `iShutdown = 1` | workers / WaitAllDone / Destroy 自己 | Destroy L776 |
| `iRegistered = 1`（新 worker 就绪） | worker 自身 cond_wait 唤醒进主循环 | CreateWorker L344 |
| `iLiveThreadNum--`（worker 退出） | Destroy | WorkerThread L569 |
| `iActiveCallers--`（AddTask 出栈） | Destroy | AddTask/Timeout/Try 出口 |
| `iBusyThreadNum--` / `iTaskQueueLen--`（任务完成） | WaitAllDone | WorkerThread 收尾 |
| `iExpanding = 0`（扩容结束） | 后续 AddTask 判断是否再扩 | CreateWorker 各出口 |

### 3.4 无死锁保证

- 单锁：不会锁自身
- ThreadQueue 锁与 pool 锁不嵌套（PutMsg/GetMsg 前必先 unlock pool）
- 无 A 等 B、B 等 A 结构（cond 等待条件全部由 broadcast 打破）

## 四、创建/销毁全流程

### 4.1 ThreadPoolCreate 顺序

```
1) 参数校验（iMinNum > 0, iMaxNum >= iMinNum, iQueueMaxSize > 0）
2) malloc(T_ThreadPoolHandle) + memset 0
3) 复制 tConfig / iIdleTimeoutMs（负数或垃圾值 → 0，禁用缩容）
4) RedBlackTree_init_root(tWorkerTree)
5) INIT_LIST_HEAD(tBusyList)
6) pthread_mutex_init(tMutex)
7) pthread_condattr_setclock(CLOCK_MONOTONIC) + pthread_cond_init(tCond)
8) ThreadQueueAPI_CreateMsg(ptTaskQueue, iQueueMaxSize, TaskReleaseCallback)
9) uiMagic = THREADPOOL_MAGIC   ← 此后 Validate 可通过
10) for i in [0, iMinNum): ThreadPool_CreateWorker(pool)
      若失败：iShutdown=1、CloseMsg、等已建 worker 退出、销毁 cond/mutex/queue、free、return NULL
11) return pool
```

**关键顺序点**：`uiMagic` 必须在 mutex/cond/queue 都初始化后设置，否则可能有其它线程通过 Validate 但访问未初始化的锁。

### 4.2 ThreadPoolDestroy 顺序

```
1) Validate(pool) → 失败 return -1
2) lock(tMutex)
   uiMagic = 0                   ← 后续所有 API 的 Validate 会失败
   iShutdown = 1
   cond_broadcast(tCond)         ← 唤醒 workers / WaitAllDone
   unlock(tMutex)
3) ThreadQueueAPI_CloseMsg(ptTaskQueue)  ← 唤醒阻塞在 GetMsg 的 worker
4) lock(tMutex)
   while (iActiveCallers > 0 || iLiveThreadNum > 0)
       cond_wait(tCond, tMutex)  ← 释放锁等待所有 worker 退出、AddTask 出栈
   unlock(tMutex)
5) ThreadQueueAPI_DestroyMsg(&ptTaskQueue)  ← 队列内残留任务通过 callback 释放
6) pthread_cond_destroy(tCond)
7) pthread_mutex_destroy(tMutex)
8) free(pool)
```

**关键顺序点**：
- Step 2 中 `uiMagic=0` 和 `iShutdown=1` 必须在**同一临界区**内完成。原因：通过 Validate 但未拿锁的线程仍会 lock，若这两个赋值分开，锁间隙内该线程可能 lock 成功但看到 `iShutdown==0`，从而按正常路径执行。
- Step 4 的双计数等待缺一不可：`iActiveCallers` 覆盖 AddTask 系列（可能仍在 PutMsg 中），`iLiveThreadNum` 覆盖 workers。
- Step 5 必须在 Step 4 之后：workers 消费队列时会解引用 `T_PoolTask*`，若先 DestroyMsg 会 UAF。
- Step 6/7 顺序：cond 依赖 mutex 存在，先 cond 后 mutex。

**已知问题**（`AI审查结果.md` R2-M1）：`Resize` 扩容路径未参与 `iActiveCallers` 计数，与 Destroy 并发存在竞态窗口。修复方案见 R2-M1 参考实现。

## 五、Worker 主循环

### 5.1 生命周期状态

```
     CreateWorker 创建
          │
          ▼
     ┌─────────────┐
     │  等待注册    │   iRegistered == 0，cond_wait
     │  (启动栅栏)  │
     └──────┬──────┘
            │  CreateWorker 完成插树 → iRegistered=1、broadcast
            ▼
     ┌─────────────┐
     │   IDLE      │◄─────────────────────┐
     │  (等待任务) │                       │
     └──────┬──────┘                       │
            │                              │ 任务完成
            │  GetMsg 返回一个 task         │
            ▼                              │
     ┌─────────────┐                       │
     │   BUSY      ├───────────────────────┘
     │  (执行 cb)  │
     └──────┬──────┘
            │  被标 iExiting=1 / iShutdown=1
            ▼                              │
     ┌─────────────┐                       │
     │   EXITING   │◄──────────────────────┘
     │  (清理退出) │   空闲超时 / Resize 缩容
     └──────┬──────┘
            │
            ▼
     iLiveThreadNum--, 摘树、broadcast、free node
```

### 5.2 主循环伪码

```c
void *ThreadPool_WorkerThread(void *arg)
{
    T_WorkerNode      *self = arg;
    T_ThreadPoolHandle *pool = self->ptPool;

    pthread_detach(pthread_self());

    /* --- 启动栅栏：等 CreateWorker 完成插树 --- */
    lock(pool->tMutex);
    while (!self->iRegistered)
        cond_wait(pool->tCond, pool->tMutex);
    unlock(pool->tMutex);

    /* --- 空闲时钟起点 --- */
    struct timespec tLastActive;
    clock_gettime(CLOCK_MONOTONIC, &tLastActive);

    /* --- 主循环 --- */
    while (1) {
        /* 从队列取任务，超时 IDLE_CHECK_INTERVAL_MS (=100ms) */
        T_PoolTask *task = NULL;
        int qret = ThreadQueueAPI_GetMsgTimeout(pool->ptTaskQueue, &task, 100);

        lock(pool->tMutex);

        /* 优先检查退出条件 */
        if (pool->iShutdown || self->iExiting) {
            unlock(pool->tMutex);
            if (task) { /* 极小概率：拿到任务同时被要求退出 */
                task->pTaskFunc(task->pUserArg);
                free(task);
                lock(pool->tMutex);
                pool->iTaskQueueLen--;
                pool->tStats.ulTotalTasksProcessed++;
                unlock(pool->tMutex);
            }
            break;
        }

        if (qret == 0 && task != NULL) {
            /* --- 拿到任务，进入 BUSY --- */
            self->iIsBusy = 1;
            list_add_tail(&self->tBusyEntry, &pool->tBusyList);
            pool->iBusyThreadNum++;
            if (pool->iBusyThreadNum > pool->tStats.iPeakBusyThreadNum)
                pool->tStats.iPeakBusyThreadNum = pool->iBusyThreadNum;
            unlock(pool->tMutex);

            task->pTaskFunc(task->pUserArg);      /* ← 用户回调 */
            free(task);

            lock(pool->tMutex);
            self->iIsBusy = 0;
            list_del_init(&self->tBusyEntry);
            pool->iBusyThreadNum--;
            pool->iTaskQueueLen--;
            pool->tStats.ulTotalTasksProcessed++;
            cond_broadcast(pool->tCond);           /* 唤醒 WaitAllDone */
            unlock(pool->tMutex);

            clock_gettime(CLOCK_MONOTONIC, &tLastActive);
        } else {
            /* --- 队列空/超时，检查空闲缩容 --- */
            if (pool->iIdleTimeoutMs > 0
                && pool->iLiveThreadNum > pool->tConfig.iMinNum)
            {
                struct timespec tNow;
                clock_gettime(CLOCK_MONOTONIC, &tNow);
                long elapsed_ms = /* tNow - tLastActive */;
                if (elapsed_ms >= pool->iIdleTimeoutMs) {
                    self->iExiting = 1;
                    unlock(pool->tMutex);
                    break;
                }
            }
            unlock(pool->tMutex);
        }
    }

    /* --- 退出清理 --- */
    lock(pool->tMutex);
    RedBlackTree_delete(&pool->tWorkerTree, &self->tRbNode);
    pool->iLiveThreadNum--;
    if (self->iIsBusy) {   /* 兜底：异常路径 */
        self->iIsBusy = 0;
        list_del_init(&self->tBusyEntry);
        pool->iBusyThreadNum--;
    }
    cond_broadcast(pool->tCond);                   /* 唤醒 Destroy */
    unlock(pool->tMutex);

    free((void *)self->tThreadConfig.sThreadName);
    free(self);
    return NULL;
}
```

### 5.3 关键点

- **启动栅栏（iRegistered）**：CreateWorker 完成 `RedBlackTree_insert` 后才置 `iRegistered=1`。worker 必须等这个信号，否则可能在没被插入红黑树时就退出，导致 `RedBlackTree_delete` 崩溃。
- **BUSY 期间释放锁**：`task->pTaskFunc(...)` 在锁外执行，允许多 worker 并发。
- **`iTaskQueueLen--` 在任务完成后减**：入队时 `++`，完成时 `--`。这个"镜像计数"与 ThreadQueue 内部长度**不完全同步**（任务出队但未执行完时，Queue 长度已减、镜像仍为原值），但保证了 `WaitAllDone` 判断"所有任务真正完成"这一语义。
- **空闲缩容双重条件**：`iLiveThreadNum > iMinNum` 且 `elapsed_ms >= iIdleTimeoutMs`。两个条件都在锁内检查，避免并发 worker 同时缩到 < iMinNum。
- **兜底 busy 清理**：正常路径下退出时 `iIsBusy` 一定是 0（BUSY 完成后才检查退出），但被 `pthread_cancel` 等异常打断时可能残留（一轮 STA-02）。

## 六、AddTask 系列

三个变体共享同一骨架，差别仅在入队等待策略：

| API | 队列满时行为 | 内部实现 |
|-----|--------------|----------|
| `ThreadPoolAddTask`        | 阻塞直到入队 | `ThreadQueueAPI_PutMsg`（无超时） |
| `ThreadPoolAddTaskTimeout` | 最多等 N ms | `ThreadQueueAPI_PutMsgTimeout(N)`；超时 → 返回 -2 |
| `ThreadPoolAddTaskTry`     | 立即返回 -2 | `ThreadQueueAPI_PutMsgTimeout(1)`（1ms 缓解 TOCTOU） |

### 6.1 统一骨架伪码

```c
int AddTask(pool, TaskFunc, pUserArg [, timeout])
{
    if (Validate(pool) != 0) return -1;
    if (TaskFunc == NULL)    return -1;

    /* --- 阶段 1：入口检查 + 扩容决策 + 计数递增 --- */
    lock(pool->tMutex);
    if (pool->iShutdown) { unlock; return -1; }

    int need_expand = 0;
    if (!pool->iExpanding
        && (pool->iBusyThreadNum + pool->iTaskQueueLen) >= pool->iLiveThreadNum
        && pool->iLiveThreadNum < pool->tConfig.iMaxNum)
    {
        need_expand = 1;
        pool->iExpanding = 1;   /* 单次节流 */
    }
    pool->iActiveCallers++;      /* ← UAF 防护起点 */
    unlock(pool->tMutex);

    /* --- 阶段 2：按需扩容（锁外，CreateWorker 内部自锁）--- */
    if (need_expand) ThreadPool_CreateWorker(pool);
    /* CreateWorker 内部会在各出口 clear iExpanding，此处无需处理 */

    /* --- 阶段 3：分配任务对象 --- */
    T_PoolTask *task = malloc(sizeof(T_PoolTask));
    if (!task) {
        lock; iActiveCallers--; broadcast; unlock;
        return -1;
    }
    task->pTaskFunc = TaskFunc;
    task->pUserArg  = pUserArg;

    /* --- 阶段 4：镜像计数与统计递增（BUG-1 修复：先增后入队）--- */
    lock(pool->tMutex);
    pool->iTaskQueueLen++;
    pool->tStats.ulTotalTasksSubmitted++;
    if (pool->iTaskQueueLen > pool->tStats.iPeakTaskQueueLen)
        pool->tStats.iPeakTaskQueueLen = pool->iTaskQueueLen;

    /* 二次 shutdown 检查（BUG-2 修复：与 Destroy 竞态防护）*/
    if (pool->iShutdown) {
        pool->iTaskQueueLen--;
        pool->tStats.ulTotalTasksSubmitted--;
        pool->iActiveCallers--;
        cond_broadcast(pool->tCond);
        unlock(pool->tMutex);
        free(task);
        return -1;
    }
    unlock(pool->tMutex);

    /* --- 阶段 5：真正入队（可能阻塞/超时）--- */
    int put_ret = ThreadQueueAPI_PutMsg[Timeout](ptTaskQueue, task, ...);
    if (put_ret != 0) {
        /* 入队失败，回滚计数 */
        lock(pool->tMutex);
        pool->iTaskQueueLen--;
        pool->tStats.ulTotalTasksSubmitted--;
        pool->iActiveCallers--;
        cond_broadcast(pool->tCond);
        unlock(pool->tMutex);
        free(task);
        return (put_ret == -3) ? -2 : -1;   /* -3=timeout→-2 */
    }

    /* --- 阶段 6：正常出栈 --- */
    lock(pool->tMutex);
    pool->iActiveCallers--;
    cond_broadcast(pool->tCond);
    unlock(pool->tMutex);
    return 0;
}
```

### 6.2 关键设计点

1. **BUG-1 修复顺序**：先更新 `iTaskQueueLen++`，再入队；否则 `WaitAllDone` 可能在"入队完成但镜像未更新"的窗口内看到 `iTaskQueueLen==0` 而误判完成。
2. **BUG-2 修复顺序**：`iTaskQueueLen++` 后再次检查 `iShutdown`；否则可能在 `iShutdown=1` 后仍将任务入队，Destroy 结束时任务泄漏。
3. **iExpanding 节流**：并发 3 个 AddTask 都判断需要扩容时，只有第一个真正调 CreateWorker，避免瞬间创建 3 个 worker 后队列消费上来又要缩。
4. **iActiveCallers 保护范围**：从阶段 1 递增到阶段 5/6 递减，覆盖了整个 `PutMsg` 期间的池内存访问，Destroy 会等它归零。

## 七、CreateWorker

被 `ThreadPoolCreate`（初始化 iMinNum 个）、`AddTask`（按需扩容）、`Resize`（扩容目标）三处调用。

### 7.1 伪码

```c
int ThreadPool_CreateWorker(pool)
{
    if (!pool) return -1;

    /* --- 原子化：检查 shutdown + 检查上限 + 预留槽位 --- */
    lock(pool->tMutex);
    if (pool->iShutdown)                       { unlock; return -1; }
    if (pool->iLiveThreadNum >= iMaxNum)       { unlock; return -1; }
    pool->iLiveThreadNum++;                    /* 预留 */
    unsigned int seq = pool->uiWorkerSeq++;
    unlock(pool->tMutex);

    /* --- 分配 node + 名字（锁外）--- */
    T_WorkerNode *w = malloc(sizeof(T_WorkerNode));
    if (!w) {
        lock; iLiveThreadNum--; iExpanding=0; unlock;   /* 回退 */
        return -1;
    }
    memset(w, 0, sizeof(*w));
    w->ptPool      = pool;
    w->iRegistered = 0;                        /* 关键：先 0 */
    INIT_LIST_HEAD(&w->tBusyEntry);
    RedBlackTree_init_node(&w->tRbNode);

    char *name = malloc(32);
    if (!name) { free(w); 回退计数; return -1; }
    snprintf(name, 32, "PoolWorker_%u", seq);

    w->tThreadConfig.pThreadFunc        = ThreadPool_WorkerThread;
    w->tThreadConfig.pThreadFuncUserArg = w;   /* 传自身 */
    w->tThreadConfig.sThreadName        = name;
    w->tThreadConfig.eSetAttr           = 0;   /* 默认属性 */

    /* --- 调用底层线程创建 --- */
    if (ThreadAPI_ThreadCreate(&w->tThreadConfig) != 0) {
        free(name); free(w); 回退计数; return -1;
    }
    /* 至此 worker 线程已启动，但正阻塞在 iRegistered 栅栏上 */

    /* --- 插树 + 广播（原子：锁内完成 3 件事）--- */
    lock(pool->tMutex);
    RedBlackTree_insert(&pool->tWorkerTree, &w->tRbNode, WorkerCmp, NULL);
    w->iRegistered   = 1;
    pool->iExpanding = 0;                      /* 允许下次扩容 */
    cond_broadcast(pool->tCond);               /* 唤醒 worker 主循环 */
    unlock(pool->tMutex);

    return 0;
}
```

### 7.2 关键设计点

1. **预留槽位在锁内**：`iLiveThreadNum++` 与"检查 < iMaxNum"必须原子，否则两个并发 CreateWorker 都能通过检查、双双 ++，超出 iMaxNum。
2. **worker 传自身指针**：`pThreadFuncUserArg = w`，worker 通过 `w->ptPool` 反向拿池句柄，避免全局变量。
3. **iRegistered 栅栏**：`ThreadAPI_ThreadCreate` 返回时 worker 线程已启动、进入 `ThreadPool_WorkerThread` 顶部、正 cond_wait `iRegistered`。这样 CreateWorker 有时间在插树完成后再放行 worker，保证"worker 见到自己一定在树中"。
4. **iExpanding 清零位置**：仅在 CreateWorker 的所有出口（成功 + 各失败点）清零。若 CreateWorker 崩溃或线程卡死不返回，iExpanding 会一直为 1，后续 AddTask 无法扩容——这是一个已知的边界（一轮未特别标注，实际场景中 pthread_create 失败率极低）。

## 八、Resize

### 8.1 语义

- `iNewMinNum > iCurrentLive`：**立即扩容**，同步创建 `iNewMinNum - iCurrentLive` 个 worker
- `iNewMaxNum < iCurrentLive`：**标记缩容**，遍历红黑树找空闲 worker 置 `iExiting=1`；忙碌 worker 等其完成当前任务后自然退出
- 其余情况：仅更新 `tConfig.iMinNum/iMaxNum`

### 8.2 伪码骨架

```c
int Resize(pool, iNewMin, iNewMax)
{
    if (Validate(pool)) return -1;
    if (iNewMin<=0 || iNewMax<=0 || iNewMin>iNewMax) return -1;

    lock(pool->tMutex);
    if (pool->iShutdown) { unlock; return -1; }

    pool->tConfig.iMinNum = iNewMin;
    pool->tConfig.iMaxNum = iNewMax;
    int cur = pool->iLiveThreadNum;

    /* ---- 扩容分支 ---- */
    if (iNewMin > cur) {
        int need = iNewMin - cur;
        unlock(pool->tMutex);              /* ← R2-M1：此处 UAF 窗口 */

        int created = 0;
        for (i = 0; i < need; i++)
            if (CreateWorker(pool) == 0) created++;

        if (created == 0) {
            lock; iMinNum=cur; iMaxNum=max(iNewMax,cur); unlock;
            return -1;
        }
        return 0;
    }

    /* ---- 缩容分支 ---- */
    if (iNewMax < cur) {
        int need_exit = cur - iNewMax;
        rb_node *pos;
        RedBlackTree_for_each(pos, &pool->tWorkerTree) {
            if (need_exit <= 0) break;
            T_WorkerNode *w = container_of(pos, T_WorkerNode, tRbNode);
            if (!w->iIsBusy && !w->iExiting) {
                w->iExiting = 1;
                need_exit--;
            }
        }
    }

    unlock(pool->tMutex);
    return 0;
}
```

### 8.3 已知问题（R2-M1）

扩容分支在 `unlock` 后进入 for 循环调 CreateWorker，未持有 `iActiveCallers`。若并发 Destroy 且 worker 全退（罕见但可能——Resize 想扩正是因为 live 少），Destroy 可能在 for 循环期间销毁 mutex，下一次 CreateWorker 内部 lock UAF。

**修复方案**：进入函数时 `iActiveCallers++`，各返回路径前 `iActiveCallers--` + broadcast。见 `AI审查结果.md` R2-M1。

## 九、WaitAllDone

### 9.1 完成判据

```
iBusyThreadNum == 0  AND  iTaskQueueLen == 0
```

两个字段都在 `tMutex` 保护下更新，判断条件天然原子。

### 9.2 伪码

```c
int WaitAllDone(pool, iTimeoutMs)
{
    if (Validate(pool)) return -1;

    struct timespec tStart;
    if (iTimeoutMs > 0) clock_gettime(CLOCK_MONOTONIC, &tStart);

    lock(pool->tMutex);
    while (1) {
        if (pool->iShutdown) { unlock; return -1; }
        if (pool->iBusyThreadNum == 0 && pool->iTaskQueueLen == 0) {
            unlock; return 0;
        }

        if (iTimeoutMs > 0) {
            /* 计算 elapsed，处理跨秒纳秒借位 */
            struct timespec tNow; clock_gettime(CLOCK_MONOTONIC, &tNow);
            long sec  = tNow.tv_sec  - tStart.tv_sec;
            long nsec = tNow.tv_nsec - tStart.tv_nsec;
            if (nsec < 0) { sec--; nsec += 1000000000L; }
            long elapsed = sec*1000 + nsec/1000000;
            if (elapsed >= iTimeoutMs) { unlock; return -2; }

            /* 剩余时间构造绝对 deadline，cond_timedwait */
            ...
            cond_timedwait(pool->tCond, pool->tMutex, &tAbstime);
        } else {
            cond_wait(pool->tCond, pool->tMutex);
        }
    }
}
```

### 9.3 关键点

- 完成判据里没有 `iLiveThreadNum`：即便还有 iMinNum 个 idle worker 存活，只要没任务在忙、队列空，就算完成。
- 唤醒来源：worker 完成任务后的 broadcast（第五章 5.2 末尾）；Destroy 的 broadcast（本 wait 会立即看到 iShutdown 返回 -1）。
- **L3（R2-L3）**：循环内每次都 `clock_gettime` 重新算 abstime，可优化为一次算出 tDeadline 复用（微秒级漂移，非功能问题）。

## 十、计数器账目一致性表

| 计数器 | ++ 位置 | -- 位置 | 不变式 |
|--------|---------|---------|--------|
| `iLiveThreadNum` | CreateWorker 预留（`iLiveThreadNum++`） | WorkerThread 退出清理 | 各时刻 == tWorkerTree 节点数（Live 计数包含"已插树未 return"和"未插树已预留"两个瞬态） |
| `iBusyThreadNum` | WorkerThread 拿到 task 后 | task 执行完 | 各时刻 == tBusyList 长度 |
| `iTaskQueueLen`  | AddTask 阶段 4 | WorkerThread task 完成后 | 与 ThreadQueue 内部长度**不完全同步**：出队后 - 完成前的窗口内，队列长度已减、iTaskQueueLen 未减 |
| `iActiveCallers` | AddTask 系列 阶段 1 | AddTask 系列 阶段 5/6 或失败回滚 | 反映"正在 PutMsg 或已 PutMsg 未收尾的调用者数" |
| `iExpanding`     | AddTask 阶段 1（决定扩容时） | CreateWorker 各出口 | 至多为 1；用作单次扩容节流 |
| `uiWorkerSeq`    | CreateWorker 内 | 从不 -- | 单调递增，用于 worker 名字唯一 |
| `tStats.ulTotalTasksSubmitted` | AddTask 阶段 4 | 失败回滚时 -- | == 累计提交成功的任务数 |
| `tStats.ulTotalTasksProcessed` | WorkerThread task 完成后 | 从不 -- | <= ulTotalTasksSubmitted |
| `tStats.iPeakBusyThreadNum`    | ++ 时若 > 记录则更新 | 从不 -- | == 历史最大 iBusyThreadNum |
| `tStats.iPeakTaskQueueLen`     | ++ 时若 > 记录则更新 | 从不 -- | == 历史最大 iTaskQueueLen |

## 十一、竞态与 UAF 防护

### 11.1 已解决

1. **Destroy 双重销毁 / 野指针**：`uiMagic` + `iShutdown` 同临界区原子设置
2. **AddTask 与 Destroy 竞态**：`iActiveCallers` 双计数，Destroy 等归零
3. **AddTask 与 AddTask 扩容竞态**：`iExpanding` 单次节流 + CreateWorker 内锁内 `iLiveThreadNum++` 检查上限
4. **CreateWorker 与 WorkerThread 启动竞态**：`iRegistered` 栅栏 + 插树在锁内
5. **WaitAllDone 与任务完成竞态**：镜像 `iTaskQueueLen` 与 `iBusyThreadNum` 同锁保护
6. **CLOCK_REALTIME 跳变**：所有 cond/超时用 `CLOCK_MONOTONIC`
7. **BUG-1（入队与镜像更新窗口）**：先 `iTaskQueueLen++` 再入队
8. **BUG-2（AddTask 与 Destroy 入队后竞态）**：入队前二次检查 `iShutdown`

### 11.2 已知未修

1. **R2-M1 Resize 扩容 UAF**：Resize 扩容路径未参与 `iActiveCallers`，与 Destroy 并发存在竞态窗口。修复计划见 AI审查结果.md。
2. **R2-M2 查询类 API 与 Destroy 并发**：`BusyThreadNumGet` / `LiveThreadNumGet` / `TaskQueueLenGet` / `StatsGet` 在 Validate→Lock 之间与 Destroy 有 UB 窗口。当前依赖时序隐式保证（Destroy 等 iLiveThreadNum→0 需 ≥100ms）。文档已在 readme.md 与 R2-M2 中标注。
3. **STA-01 Destroy 无超时**：若 task 死循环，Destroy 永远等不到 iActiveCallers/iLiveThreadNum 归零。契约：用户 task 必须有限执行。
4. **STA-02 task 异常退出**：`pthread_cancel` / `pthread_exit` 会跳过 iBusyThreadNum-- 与 free(task)。契约：task 只能 return。

### 15.1 内部辅助函数

#### 15.1.1 `ThreadPool_Validate`

**位置**：`Pool.c:82`
**签名**：`static inline int ThreadPool_Validate(T_ThreadPoolHandle *h)`
**类型**：内部；被几乎所有公共 API 入口调用
**线程安全**：读 `uiMagic` 无锁，依赖 `Destroy` 在同一临界区内清零 uiMagic + 设置 iShutdown 的原子性

**参数**：
- `h`：待校验的池句柄

**返回**：
- `0`：句柄有效（非 NULL 且 `uiMagic == THREADPOOL_MAGIC`）
- `-1`：NULL 或 magic 不匹配（含重复销毁 / 野指针）

**逻辑顺序**：
1. 若 `h == NULL` → 返回 -1
2. 若 `h->uiMagic != 0x5448504C` → 返回 -1
3. 否则返回 0

**并发点**：
- 无锁读 `uiMagic`。此设计接受"Validate→Lock 窗口 UAF"的软保证（详见第十一章 R2-M2）
- Destroy 在锁内 `uiMagic=0` 与 `iShutdown=1` 同临界区，保证通过 Validate 的调用者若能进入锁必看到 `iShutdown==1`

---

#### 15.1.2 `ThreadPool_TaskReleaseCallback`

**位置**：`Pool.c:115`
**签名**：`static void ThreadPool_TaskReleaseCallback(void *pData)`
**类型**：内部；作为 callback 传给 `ThreadQueueAPI_CreateMsg`
**线程安全**：由 ThreadQueue 内部同步保证

**用途**：Destroy 时 `ThreadQueueAPI_DestroyMsg` 内部对残留 `T_PoolTask*` 元素逐条调此函数释放。

**逻辑**：
1. `pData` 非 NULL 则 `free(pData)`

**注意**：
- 仅释放 `T_PoolTask` 结构体，不释放 `pUserArg`（用户数据生命周期由用户管）
- 不递减 `iTaskQueueLen`：Destroy 已经不再依赖此计数

#### 15.1.3 `ThreadPool_WorkerCmp`

**位置**：`Pool.c:140`
**签名**：`int ThreadPool_WorkerCmp(struct rb_node *a, struct rb_node *b, void *pArg)`
**类型**：内部；作为比较函数传给 `RedBlackTree_insert/search/delete`

**参数**：
- `a`、`b`：红黑树节点（对应 `T_WorkerNode.tRbNode`）
- `pArg`：未用

**返回**：`-1 / 0 / +1`，按 `(unsigned long)pthread_t` 数值比较

**逻辑**：
1. 用 `RedBlackTree_entry` 从 `rb_node*` 反查宿主 `T_WorkerNode*`
2. 将 `tThreadConfig.tThreadPid` 转 `unsigned long` 数值比较

**注意**：
- 依赖 Linux `pthread_t == unsigned long`（IMX6ULL glibc 保证）；移植到其它 pthread 实现需重写
- 相同 pthread_t 返回 0，红黑树认为是同一节点

#### 15.1.4 `ThreadPool_FindWorker`

**位置**：`Pool.c:168`
**签名**：`T_WorkerNode *ThreadPool_FindWorker(T_ThreadPoolHandle *ptPool, T_ThreadCreateConfig *ptThreadConfig)`
**类型**：内部辅助（V1.1 未在其它函数中使用，保留供扩展）

**参数**：
- `ptPool`：池句柄
- `ptThreadConfig`：查询键，只用 `tThreadPid`

**返回**：`T_WorkerNode*` 或 `NULL`

**逻辑顺序**：
1. 校验非 NULL
2. 栈上构造临时 `T_WorkerNode tKey`，仅填 `tThreadConfig.tThreadPid`
3. `RedBlackTree_search` → O(log n)
4. 从 `rb_node*` 反查 `T_WorkerNode*`

**并发点**：**函数本身不加锁**，调用者必须在 `tMutex` 内调用。当前 V1.1 无调用点。

#### 15.1.5 `ThreadPool_CreateWorker`

**位置**：`Pool.c:210`（详细伪码见第七章）
**签名**：`int ThreadPool_CreateWorker(T_ThreadPoolHandle *ptPool)`
**类型**：内部；被 `ThreadPoolCreate`（初始化 iMinNum 个）、`AddTask*`（按需扩容）、`Resize`（扩容目标）三处调用

**返回**：
- `0`：worker 创建成功、已插树、`iRegistered=1`
- `-1`：任一环节失败（含 `iShutdown` 已置、已达 iMaxNum、malloc 失败、pthread_create 失败）

**逻辑顺序**（简版；详版见第七章）：
1. 锁内检查 `iShutdown`、`iLiveThreadNum < iMaxNum`，**原子预留** `iLiveThreadNum++`、取 `uiWorkerSeq++`
2. 解锁后分配 `T_WorkerNode` + 32B 名字缓冲，初始化字段（`iRegistered=0` 是关键）
3. 调 `ThreadAPI_ThreadCreate` 拉起 worker 线程；worker 立即进入 `WorkerThread` 顶部并阻塞在 `iRegistered` 栅栏
4. 锁内 `RedBlackTree_insert`、`iRegistered=1`、`iExpanding=0`、`broadcast`

**边界与错误**：
- 任一失败点都必须回退 `iLiveThreadNum` 与 `iExpanding`；已 free 已分配的 name/node
- `iExpanding` 仅由此函数清零（AddTask 只设 1），所以 CreateWorker 崩溃会永久锁死扩容

**并发点**：
- 步骤 1/4 都在 `tMutex` 内
- 步骤 3 的 `pthread_create` 与 worker 的 `iRegistered` 栅栏配合：worker 走到 `ThreadPool_WorkerThread` 时 CreateWorker 可能还未插树，栅栏保证 worker 见到自己在树中才进入主循环

**状态变更**：
- `iLiveThreadNum ++`（成功）或 `+/- 平衡`（失败）
- `uiWorkerSeq ++`（成功、失败均已递增；失败情况下"跳号"是无害的）
- `iExpanding` 清零
- `tWorkerTree` 增一节点（成功）

#### 15.1.6 `ThreadPool_WorkerThread`

**位置**：`Pool.c:396`（详细伪码见第五章）
**签名**：`void *ThreadPool_WorkerThread(void *arg)`（`arg = T_WorkerNode*`）
**类型**：worker 线程主函数；每个 worker 独立执行

**逻辑顺序**（简版；详版见第五章）：
1. `pthread_detach(pthread_self())` — 无需外部 join
2. 锁内 cond_wait `iRegistered==1`（启动栅栏）
3. 记录 `tLastActive`（空闲时钟起点）
4. 主循环：
   - `ThreadQueueAPI_GetMsgTimeout(pool->ptTaskQueue, &task, 100)`
   - 锁内：优先 check `iShutdown || iExiting` → 若命中：退出（若同时拿到 task 则先执行以避免任务丢）
   - 拿到任务：`iIsBusy=1`、加入 `tBusyList`、`iBusyThreadNum++`、更新 peak、解锁执行 `task->pTaskFunc(task->pUserArg)`、`free(task)`、锁内 `iIsBusy=0`、摘 list、`iBusyThreadNum--`、`iTaskQueueLen--`、`ulTotalTasksProcessed++`、`broadcast`、更新 `tLastActive`
   - 超时无任务：若 `iIdleTimeoutMs>0` 且 `iLiveThreadNum>iMinNum` 且 `elapsed>=idle_timeout` → 自标 `iExiting=1`、break
5. 退出清理：锁内 `RedBlackTree_delete`、`iLiveThreadNum--`、兜底 busy 清理、`broadcast`
6. 锁外 `free(sThreadName)`、`free(worker_node)`

**边界与错误**：
- 步骤 4 中 task 拿到但同时收到 shutdown：仍先执行完并递减 `iTaskQueueLen`，保证账目一致
- 步骤 5 兜底：正常路径 `iIsBusy` 一定是 0，但被 `pthread_cancel/exit` 打断时可能残留 1（STA-02 已知问题）

**并发点**：
- BUSY 段落释放锁执行 `pTaskFunc`，允许多 worker 并发
- 退出前的 `broadcast` 是 Destroy 等待 `iLiveThreadNum==0` 的唯一唤醒源

**状态变更**：
- 上任务：`iIsBusy 0→1`、`iBusyThreadNum++`、`tBusyList` 加节点
- 完成任务：反向 + `iTaskQueueLen--` + `ulTotalTasksProcessed++`
- 退出：`iLiveThreadNum--`、摘树

### 15.2 生命周期 API

#### 15.2.1 `ThreadAPI_ThreadPoolCreate`

**位置**：`Pool.c:598`（流程见第 4.1 节）
**签名**：`T_ThreadPoolHandle* ThreadAPI_ThreadPoolCreate(T_ThreadPoolConfig t_Config)`
**类型**：公共
**幂等性**：否；每次调用 malloc 新句柄

**参数**（值传入）：
- `t_Config.iMinNum > 0`
- `t_Config.iMaxNum >= iMinNum`
- `t_Config.iQueueMaxSize > 0`
- `t_Config.iIdleTimeoutMs`：0 或负值表示禁用缩容

**返回**：
- 非 NULL：池句柄
- NULL：参数无效 / malloc 失败 / mutex/cond/queue 初始化失败 / 初始 iMinNum 个 worker 创建失败

**逻辑顺序**：
1. 校验 `iMinNum/iMaxNum/iQueueMaxSize` 与大小关系
2. `malloc(T_ThreadPoolHandle) + memset(0)`
3. 复制 `tConfig`；容错 `iIdleTimeoutMs`（负数→0）
4. 初始化 `tWorkerTree`、`tBusyList`、`tMutex`
5. 用 `pthread_condattr_setclock(CLOCK_MONOTONIC)` 初始化 `tCond`
6. `ThreadQueueAPI_CreateMsg(iQueueMaxSize, TaskReleaseCallback)`
7. `uiMagic = THREADPOOL_MAGIC`（此后 Validate 通过）
8. `for i in [0, iMinNum)` 调 `CreateWorker`；任一失败则：`iShutdown=1`、`CloseMsg`、等已建 worker 退出、销毁 cond/mutex/queue、`free(pool)`、return NULL
9. return pool

**边界与错误**：
- 步骤 7 顺序关键：`uiMagic` 必须在同步原语初始化完成后设置，避免其它线程 Validate 通过但访问未初始化的锁
- 步骤 8 rollback 路径要等 worker 退出后才 free pool，避免 UAF

**并发点**：调用前无并发（用户单线程调用）；rollback 中依赖 worker 主循环的 `iLiveThreadNum--` broadcast

**状态变更**：`uiMagic`、全部字段初值、`iLiveThreadNum` = 初始成功创建数（正常 = iMinNum）

---

#### 15.2.2 `ThreadAPI_ThreadPoolDestroy`

**位置**：`Pool.c:748`（流程见第 4.2 节）
**签名**：`int ThreadAPI_ThreadPoolDestroy(T_ThreadPoolHandle *pt_ThreadPoolHandle)`
**类型**：公共

**返回**：
- `0`：销毁完成
- `-1`：Validate 失败（重复销毁 / 野指针）

**逻辑顺序**：
1. Validate → 失败 -1
2. 锁内 `uiMagic=0`、`iShutdown=1`、broadcast、解锁（**同一临界区完成前两个赋值**）
3. `ThreadQueueAPI_CloseMsg(ptTaskQueue)` 唤醒 `GetMsg` 阻塞的 worker
4. 锁内 `while(iActiveCallers>0 || iLiveThreadNum>0) cond_wait`
5. `ThreadQueueAPI_DestroyMsg(&ptTaskQueue)`（会用 `TaskReleaseCallback` 释放残留 task）
6. `pthread_cond_destroy(tCond)`
7. `pthread_mutex_destroy(tMutex)`
8. `free(pool)`

**边界与错误**：
- 步骤 2 的原子性：确保通过 Validate 的调用者 lock 后一定看到 `iShutdown==1` 并放弃操作
- 步骤 4 无超时：若 task 死循环、Destroy 挂死（STA-01）
- 步骤 5 必须先于 6/7：workers 消费队列时会解引用 `T_PoolTask*`

**并发点**：
- 步骤 2 的 broadcast 唤醒：workers 正 `GetMsg`（未必唤醒，配合步骤 3 的 CloseMsg）、`WaitAllDone` cond_wait
- 步骤 4 的等待被 worker 退出、AddTask 出栈的 broadcast 唤醒

**状态变更**：`uiMagic=0`、`iShutdown=1`、所有 worker 退出、全部内存释放

**已知问题**：R2-M1 Resize 扩容路径未参与 `iActiveCallers`，与 Destroy 并发有 UAF 窗口

#### 15.3.1 `ThreadAPI_ThreadPoolAddTask`

**入队策略**：`ThreadQueueAPI_PutMsgTimeout(iTimeoutMs)`

#### 15.3.3 `ThreadAPI_ThreadPoolAddTaskTry`

**位置**：`Pool.c:1260`
**签名**：`int ThreadAPI_ThreadPoolAddTaskTry(T_ThreadPoolHandle *pool, ThreadFunctionType TaskFunc, void *pUserArg)`
**入队策略**：`ThreadQueueAPI_PutMsgTimeout(1)`（1ms 缓解 TOCTOU）

**返回**：

- `-1`：Validate/TaskFunc/malloc/shutdown 失败
- `-2`：队列满

**关键差异点**：
- **多一步"快速检查"**（阶段 2.5）：锁内查 `iTaskQueueLen >= iQueueMaxSize` → 立即返回 `-2`（无需分配 task）

**逻辑顺序**（差异部分）：
1. 阶段 1（含 `iActiveCallers++`）
2. **阶段 2.5**：锁内查队列长度；满则 `iActiveCallers--` + broadcast + return -2
3. 阶段 3-4（malloc、镜像 ++、shutdown 双检）
4. 阶段 5：`PutMsgTimeout(task, 1)`；超时 → return -2
5. 阶段 6

### 15.4 运行控制 API

#### 15.4.1 `ThreadAPI_ThreadPoolWaitAllDone`

**位置**：`Pool.c:1026`（伪码见第九章）
**签名**：`int ThreadAPI_ThreadPoolWaitAllDone(T_ThreadPoolHandle *pool, int iTimeoutMs)`

**返回**：
- `0`：所有任务完成
- `-1`：Validate 失败 / 等待中检测到 `iShutdown`
- `-2`：等待超时（仅 `iTimeoutMs>0` 时可能）

**参数**：
- `iTimeoutMs > 0`：绝对超时；`<= 0`：无限等（`cond_wait`）

**逻辑顺序**：
1. Validate
2. `iTimeoutMs>0` 则记录 `tStart = CLOCK_MONOTONIC 当前`
3. 锁内循环：
   - `iShutdown` → return -1
   - `iBusyThreadNum==0 && iTaskQueueLen==0` → return 0
   - 有超时：算 elapsed（处理跨秒纳秒借位）；`>= iTimeoutMs` return -2；否则算剩余 deadline `cond_timedwait`
   - 无超时：`cond_wait`
4. 循环回到第 3

**边界与错误**：
- 完成判据同时看 `iBusyThreadNum` 和 `iTaskQueueLen`；两个字段同锁保护
- 与 Destroy 并发时：Destroy 的 broadcast + `iShutdown=1` 让 WaitAllDone 立即返回 -1
- 循环内每次 `clock_gettime` 重取 abstime 有微秒漂移（R2-L3，可优化为一次算 deadline）

**并发点**：worker 完成任务后的 broadcast、Destroy 的 broadcast、Resize 的 broadcast（无害唤醒）

**状态变更**：无（只读）

---

#### 15.4.2 `ThreadAPI_ThreadPoolResize`

**位置**：`Pool.c:1402`（伪码见第八章）
**签名**：`int ThreadAPI_ThreadPoolResize(T_ThreadPoolHandle *pool, int iNewMinNum, int iNewMaxNum)`

**返回**：
- `0`：Resize 完成（含"部分扩容成功"）
- `-1`：参数无效、`iShutdown`、**全部**扩容失败

**参数约束**：`iNewMin>0 && iNewMax>0 && iNewMin<=iNewMax`

**逻辑顺序**：
1. Validate + 参数校验
2. 锁内 `iShutdown` 检查
3. 更新 `tConfig.iMinNum/iMaxNum`；快照 `iCurrentLive = iLiveThreadNum`
4. **扩容分支** (`iNewMin > iCurrentLive`)：
   - 解锁
   - for 循环调 `CreateWorker`，统计成功数
   - 若全失败：锁内回滚 `iMinNum=iCurrentLive`、`iMaxNum=max(iNewMax,iCurrentLive)`；return -1
   - 若部分/全部成功：return 0（**FUN-01**：不通过返回码告知调用者部分失败）
5. **缩容分支** (`iNewMax < iCurrentLive`)：
   - 锁内遍历 `tWorkerTree`，找 `!iIsBusy && !iExiting` 的 worker 置 `iExiting=1`，直到标够 `iCurrentLive - iNewMax` 个
   - 忙碌 worker 完成当前任务后自然退出（主循环会检测 `iExiting`）
   - 若没标够（都忙）：仅日志提示；剩余会随空闲缩容或后续 Resize 处理
6. 解锁 return 0

**边界与错误**：
- **FUN-01**：部分扩容失败仍返回 0，调用者无法感知
- **FUN-02**：回滚 `iMinNum=iCurrentLive` 时，若创建过程中有 worker 空闲缩容，回滚值可能偏高
- **R2-M1（未修）**：扩容分支解锁后调 `CreateWorker`，未参与 `iActiveCallers`，与 Destroy 并发有 UAF 窗口

**并发点**：
- 扩容分支：解锁-CreateWorker 循环期间池锁短暂空闲
- 缩容分支：遍历红黑树时全程持锁

**状态变更**：
- `tConfig.iMinNum/iMaxNum`
- 扩容：`iLiveThreadNum` 增加、`tWorkerTree` 增节点
- 缩容：目标 worker 的 `iExiting=1`；worker 实际退出发生在其主循环下次醒来

### 15.7 调试与排错索引

按"症状 → 排查函数 / 字段"：

| 症状 | 首查 | 次查 |
|------|------|------|
| Destroy 挂死 | `iActiveCallers`（AddTask 未 --）/ `iLiveThreadNum`（worker 未退出） | task 是否死循环 |
