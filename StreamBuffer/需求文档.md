# 需求文档 — StreamBuffer 流缓冲区（通用字节流环形缓冲库）

> **定位**：通用字节流环形缓冲库。核心是"生产者攒字节流 + 消费者按阈值/超时批量出队"。
> **日志写文件**是其典型应用之一；未来可扩展串口/网络攒发等任何"攒数据 + 批量处理"场景。
> 目标平台：IMX6ULL (ARM Linux) | 作者：zlzksrl | 草案日期：2026-07-09
> 状态：**需求已定稿**（设计决策见第五节）

---

## 一、原始需求（以日志写文件为引子，库本身通用）

1. 用线程队列实现"写文件 / 日志"功能。
2. 需要打印日志的地方：**格式化好字符串后入队**。
3. **出队的地方写文件**。
4. 例如写 CSV：数据记录完要关闭文件时，需**先把队列内容全部写入文件再关闭**，且**关闭后不允许再入队**。
5. 写文件**不是来一条写一条**，而是**累积到一定条数，或定时写一次**，减小硬盘磨损。
6. **入队最好不要有阻塞**；出队线程优先级低，写文件不影响主进程。
7. **减少拷贝、尽量减少 malloc**（日志会写很多）。

> 经讨论明确：做成**通用库**（不只日志）；**消费线程由用户自行创建**，库只提供"缓冲 + 触发等待 + 出队"。

---

## 二、需求理解：流式字节缓冲 + 用户消费线程出队

```
业务线程(多个)                        用户消费线程(自建, 优先级自定)
   │ 写字节流(变长)                       │
   ▼                                     │
 PutData(buf,len) ──→ [字节流环形缓冲]    │
                     (预分配, 变长紧凑,     │  Wait(等 used≥阈值/超时/关闭)
                      环形复用, 零malloc)   │  GetData/GetDataAddress/回调 出队
                                         ▼
                                   用户自行消费(fwrite/send/write串口…)
```

- **生产者（业务线程）**：`PutData` 写字节流（变长）→ 紧凑追加进环形缓冲。**快**：不阻塞、不 malloc。
- **消费者（用户自建线程）**：`Wait` 阻塞等触发 → 出队（GetData/GetDataAddress/回调）→ 自行消费。
- **解耦**：库不创建线程，只提供缓冲与出队原语；数据产生与下游 I/O 分离。
- **优雅关闭**：`Close` 阻止写入 + broadcast 唤醒 `Wait`；用户线程取空剩余后退出；可 `Reopen` 恢复。

## 三、现有库能力分析

| 需求点 | ThreadQueue | WindowQueue | 结论 |
|--------|-------------|-------------|------|
| **写入不阻塞** | 满则阻塞 ✗ | 满则丢老 | 都不完美 |
| **少 malloc** | ✗ void* 每条 malloc | ✓ 值拷贝预分配 | WindowQueue 胜 |
| **变长紧凑无浪费** | ✗ 每条一指针 | ✗ 定长截断/浪费 | 都不够 |
| **触发等待 + 出队** | ✓ Get 带超时 | ✗ | ThreadQueue 思路对路 |
| **优雅关闭排空** | ✓ | ✗ | ThreadQueue 胜 |

**结论**：无单一现成库满足。→ **新增 StreamBuffer 库**，核心采用**字节流环形缓冲区**，提供三种消费方式（GetData 拷贝 / GetDataAddress 零拷贝逐段 / 回调零拷贝自动），线程由用户管。

## 四、功能规格

### 4.1 写入 `PutData(buf, len)`（生产者）
- 入参：字节串 + 长度（变长；日志场景下用户已格式化好、含 `\r\n`）。
- 紧凑追加到环形缓冲 write 指针处（一次或两次 memcpy，无 malloc）。
- **跨回绕边界**：若 `write+len > capacity`，**分两段 memcpy**（write 到尾 + 头到剩余），保证变长紧凑、零浪费。
- 写入后若 `used ≥ 阈值`，**signal 唤醒一个**阻塞在 Wait 的消费者。
- **满则丢新**（D1）：剩余空间 `capacity-used < len` → 丢弃本段（不入队）、统计 `dropped`、立即返回、**不阻塞**。
- **单段上限**：`len > capacity` 时按容量截断。
- Close 后调用返回 -2。

### 4.2 消费（用户自建线程；三种方式，可配合）
- 库**不创建线程**。Wait 返回值为 `StreamBufferStatus` 枚举（统一 Wait 返回与回调触发状态）：
  - **>0 一律有数据**：`TRIGGER(2)`=达阈值/被唤醒、`TIMEOUT_DATA(1)`=超时有数据、`CLOSE_DATA(3)`=关闭有数据；
  - `<=0`：`TIMEOUT_EMPTY(0)`=超时空、`FLUSH_EMPTY(-1)`=Flush空、`CLOSE_EMPTY(-2)`=关闭空、`INVALID(-3)`=参数无效、`NOINIT(-4)`=未初始化。
- **方式A（GetData 拷贝式）**：`r=Wait(...,&used); if(r>0) while(GetData(...)>0) 消费; if(r<=-2) break;` —— 合并回绕两段，**多消费者安全**。
- **方式A'（GetDataAddress 零拷贝，逐段取）**：`n=GetDataAddress(&ptr, max)` 输出**本段**数据地址（二级指针，不拷贝），回绕不合并（只本段），消费 `min(本段连续长度, max)`，返回消费量。**⚠️ 地址仅在下次操作前有效，须立即使用；建议单消费者。**
- **方式B（零拷贝回调，可选）**：`SetConsumeCallback(sb, cb, ctx)` 注册后，`Wait` 返回 >0 时先阻塞调 `cb(status, data, len, ctx)`（回绕分两次，每段 len=该连续段长度，data+len 不越界）。回调返回值**钳位到 [0,len]**（<0→0、>len→len，并打印警告）。库偏移推进 read，按"剩余 used + 关闭状态"**重算返回码与 used**（剩余>0+关闭→CLOSE_DATA；剩余>0+未关闭→TRIGGER；剩余==0+关闭→CLOSE_EMPTY；剩余==0+未关闭→TIMEOUT_EMPTY）。**建议单消费者。**
- 三种方式**设计上支持配合**（如回调消费大部分 + GetData 取零头），**但实际应用通常只用其中一种**。

### 4.3 触发与唤醒
- `Wait` 的触发：`used ≥ iFlushBytes` OR `超时 timeo` OR `Close`。
- **唤醒策略**：`PutData`/`Flush` 用 **signal**（唤醒一个）；`Close` 用 **broadcast**（唤醒全部）。
- **timeo=0**：不等待，立即按当前状态计算返回（used>0→TRIGGER；used==0 且未关闭→TIMEOUT_EMPTY；已关闭按数据→CLOSE_DATA/CLOSE_EMPTY）。
- 阈值来自配置 `iFlushBytes`（必须 ≤ iCapacity），超时由 `Wait` 参数传入。

### 4.4 优雅关闭与重开（核心场景，需求 4）
```
Close()                ← 阻止新写入（PutData 返回 -2）+ broadcast 唤醒所有 Wait
      │                        （不清空缓冲，剩余仍可取）
用户消费线程：
   Wait 返回 CLOSE_DATA(3) → GetData/GetDataAddress 取数据
   → ... 取空后 Wait 返回 CLOSE_EMPTY(-2) → 退出线程
主线程：
   pthread_join(消费线程) → Destroy()
```
- 保证：关闭后排空，缓冲无残留，下游数据完整。
- **Reopen（Close 可逆）**：`Reopen()` 重置关闭标志、恢复 PutData（不清空，保留剩余）；前置：已处于关闭状态。
- **幂等**：Close 重复调用返回 0；Destroy 重复（`*pp==NULL`）返回 -1。

### 4.5 内存模型：字节流环形缓冲区（D2）
- Init 预分配一片连续内存（容量 `iCapacity`），全程复用，**零 malloc**。
- `read`/`write`/`used` 三量环形管理；**容量强制为 2 的幂**，用 `&mask` 高效回绕（非 2 的幂 Init 报错）。
- 变长字节紧凑存储，无空间浪费。

### 4.6 Init 校验规则
- `*pp` 必须为 NULL（防重复初始化）。
- `iCapacity > 0` 且**必须为 2 的幂**（否则报错）。
- `iFlushBytes > 0` 且 `<= iCapacity`（阈值大于容量不允许）。
- 不满足任一返回 -1。

### 4.7 并发与多消费者
- **建议单消费者**。多消费者场景**请用 GetData（拷贝式）**，避免回调/GetDataAddress 的内部地址被多线程并发持有导致覆盖竞争。
- **PutData 与 GetDataAddress 并发风险**：GetDataAddress 返回的内部地址，推进 read 后即可被新的 PutData 覆盖；用户须保证使用该地址期间本队列无 PutData 写入该段（单消费者+用完即弃可满足）。
- **轮询消费者**：不使用 Wait、仅用 GetData/GetDataAddress 轮询的消费者，需自行定期检查 `IsClosed()` 响应关闭。
- 全程互斥锁保护；回调/Wait 内持锁，回调须快速返回、禁调本队列 API。

## 五、关键设计决策（已定稿 ✓）

| 编号 | 决策 | 选定 |
|------|------|------|
| **D1** | 缓冲满策略 | 有界 + 满则丢新（不阻塞，丢最新，统计丢弃数）✓ |
| **D2** | 内存模型 | 字节流环形缓冲区（预分配、变长紧凑、环形复用、零 malloc）✓ |
| **D3** | 触发与唤醒 | used≥阈值 / 超时 / Close；PutData/Flush 用 signal，Close 用 broadcast；timeo=0 立即返回 ✓ |
| **D4** | 消费方式 | 三种：GetData(拷贝) / GetDataAddress(零拷贝逐段) / 回调(零拷贝自动)，可配合 ✓ |
| **D5** | 消费线程 | 由用户创建（线程数/优先级/消费方式自定，库不创建线程）✓ |
| **D6** | 库名/前缀 | StreamBuffer（`StreamBufferAPI_*`）✓ |
| **D7** | 关闭可逆 | Close 可逆，提供 Reopen（重置关闭标志，保留数据）✓ |
| **D8** | 容量约束 | iCapacity 强制 2 的幂（mask 回绕）；iFlushBytes ≤ iCapacity，Init 校验 ✓ |
| **D9** | 多消费者 | 建议单消费者；多消费者用 GetData 拷贝式（回调/GetDataAddress 不建议多线程）✓ |
| **D10** | 回调返回值 | 钳位到 [0,len]（越界打印警告）；Close/Destroy 幂等 ✓ |

## 六、典型应用

- **日志写文件**：用户消费线程出队后 `fwrite` 到自己 `fopen` 的 `FILE*`（CSV 等）。
- **串口/网络攒发**：用户消费线程出队后 `write`/`send` 一批字节，减少系统调用次数。
- **任何"攒字节流 + 定量/定时批量处理"场景**。
