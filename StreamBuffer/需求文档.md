# 需求文档 — StreamBuffer 流缓冲区（通用字节流环形缓冲库）

> **定位**：通用字节流环形缓冲库。核心是"生产者攒字节流 + 消费者按阈值/超时批量出队"。
> **日志写文件**是其典型应用之一；未来可扩展串口/网络攒发等任何"攒数据 + 批量处理"场景。
> 目标平台：IMX6ULL (ARM Linux) | 作者：zlzksrl | 草案日期：2026-07-09
> 状态：**需求已定稿**（设计决策见第五节）

---

## 一、原始需求（以日志写文件为引子，库本身通用）

1. 用线程队列实现"写文件 / 日志"功能。
2. 需要打印日志的地方：**格式化好字符串后入队**。
3. **出队的地方写文件**。
4. 例如写 CSV：数据记录完要关闭文件时，需**先把队列内容全部写入文件再关闭**，且**关闭后不允许再入队**。
5. 写文件**不是来一条写一条**，而是**累积到一定条数，或定时写一次**，减小硬盘磨损。
6. **入队最好不要有阻塞**；出队线程优先级低，写文件不影响主进程。
7. **减少拷贝、尽量减少 malloc**（日志会写很多）。

> 经讨论明确：做成**通用库**（不只日志）；**消费线程由用户自行创建**，库只提供"缓冲 + 触发等待 + 出队"。

---

## 二、需求理解：流式字节缓冲 + 用户消费线程出队

```
业务线程(多个)                        用户消费线程(自建, 优先级自定)
   │ 写字节流(变长)                       │
   ▼                                     │
 PutData(buf,len) ──→ [字节流环形缓冲]    │
                     (预分配, 变长紧凑,     │  Wait(等 used≥阈值/超时/关闭)
                      环形复用, 零malloc)   │  GetData(buf,max) 出队
                                         ▼
                                   用户自行消费(fwrite/send/write串口…)
```

- **生产者（业务线程）**：`PutData` 写字节流（变长）→ 紧凑追加进环形缓冲。**快**：不阻塞、不 malloc。
- **消费者（用户自建线程）**：`Wait` 阻塞等触发（used≥阈值 / 超时 / Close）→ `GetData` 非阻塞出队当前可用字节 → 用户自行消费（fwrite/send 等）。线程数、优先级、消费方式全由用户决定。
- **解耦**：库不创建线程，只提供缓冲与出队原语；数据产生与下游 I/O 分离。
- **优雅关闭**：`Close` 阻止写入 + 唤醒 `Wait`；用户线程把剩余 `GetData` 取空后退出。

## 三、现有库能力分析

| 需求点 | ThreadQueue | WindowQueue | 结论 |
|--------|-------------|-------------|------|
| **写入不阻塞** | 满则阻塞 ✗ | 满则丢老 | 都不完美 |
| **少 malloc** | ✗ void* 每条 malloc | ✓ 值拷贝预分配 | WindowQueue 胜 |
| **变长紧凑无浪费** | ✗ 每条一指针 | ✗ 定长截断/浪费 | 都不够 |
| **触发等待 + 出队** | ✓ Get 带超时 | ✗ | ThreadQueue 思路对路 |
| **优雅关闭排空** | ✓ | ✗ | ThreadQueue 胜 |

**结论**：无单一现成库满足。→ **新增 StreamBuffer 库**，核心采用**字节流环形缓冲区**：
- 变长字节紧凑存储、环形复用、零 malloc；
- 满则丢新、不阻塞；
- `Wait`（按字节阈值/超时/关闭触发）+ `GetData`（出队）两步式消费，线程由用户管；
- `Close` 唤醒等待者，剩余可继续 `GetData` 取空，保证优雅关闭。

## 四、功能规格

### 4.1 写入 `PutData(buf, len)`（生产者）
- 入参：字节串 + 长度（变长；日志场景下用户已格式化好、含 `\r\n`）。
- 紧凑追加到环形缓冲的 `write` 指针处（一次 memcpy，无 malloc）。
- 写入后若 `used ≥ 阈值`，唤醒一个阻塞在 `Wait` 的消费者。
- **满则丢新**（D1）：剩余空间 < 本段长度 → 丢弃本段（不入队）、统计 `dropped`、立即返回、**不阻塞**。
- **单段上限**：超过缓冲容量时按容量截断。

### 4.2 消费（用户自建线程，`Wait` + `GetData`）
- 库**不创建线程**。用户自建消费线程，循环：
  1. `Wait(timeo, &used)`：阻塞等 `used ≥ 阈值` / `超时 timeo` / `Close`。
     **返回值 >0 一律表示有数据**（2=阈值触发 / 1=超时有数据 / 3=关闭有数据）；
     `0`=超时无数据；`-1`=被 Flush 唤醒无数据；`-2`=关闭无数据；`-3`=参数无效；`-4`=未初始化。
     经 `used` 输出当前未消费字节数。
  2. `GetData(buf, max)`：非阻塞出队 `min(max, used)` 字节（环形回绕按两段拷贝），循环取空。
  3. 取到的字节由用户自行消费（fwrite / send / write 串口 / …）。
- 典型循环：`r=Wait(...,&used); if(r>0) while(GetData(...)>0) 消费; if(r<=-2) break;`
- **可选：零拷贝消费回调**（特殊场景）：`SetConsumeCallback(sb, cb, ctx)` 注册后，`Wait` 触发且
  `used>0` 时，在返回前先阻塞调用 `cb(status, data, len, ctx)`（零拷贝传内部指针，回绕分两次），
  按返回值偏移推进 read、剩余量经 `used` 返回，替代 GetData。`status` 为 `STREAMBUFFER_CB_FLUSH` /
  `_THRESHOLD` / `_CLOSE`。回调持锁执行，须快速返回（阻塞但不耗时）、禁调本队列 API。
- 线程数、优先级、消费方式由用户决定。

### 4.3 触发批量
- `Wait` 的触发：`used ≥ 字节阈值`（如 4KB）OR `超时 timeo`（由用户传入）OR `Close`。
- 阈值来自配置（`iFlushBytes`），超时由 `Wait` 参数传入（灵活，用户按需）。
- `Flush` 可主动唤醒等待者（不等阈值/超时）。

### 4.4 优雅关闭（核心场景，需求 4）
```
Close()                ← 阻止新数据写入（PutData 返回 -2）+ 唤醒 Wait（返回 -2）
      │                        （不清空缓冲，剩余仍可 GetData 取出）
用户消费线程：
   Wait 返回 -2 → 循环 GetData 取空剩余 → 退出线程
主线程：
   pthread_join(消费线程) → Destroy()
```
保证：关闭后缓冲**无残留**，下游（如文件）数据**完整**。

### 4.5 内存模型：字节流环形缓冲区（D2）
- Init 预分配一片连续内存（容量 `iCapacity` 可配，如 64KB），全程复用，**零 malloc**。
- `read`/`write`/`used` 三量环形管理（容量建议为 2 的幂，用 mask 高效回绕）。
- 变长字节紧凑存储，短数据占少、长数据占多，**无空间浪费**。

## 五、关键设计决策（已定稿 ✓）

| 编号 | 决策 | 选定 |
|------|------|------|
| **D1** | 缓冲满策略 | **有界 + 满则丢新**（不阻塞，丢最新数据，统计丢弃数）✓ |
| **D2** | 内存模型 | **字节流环形缓冲区**（预分配、变长紧凑、环形复用、零 malloc）✓ |
| **D3** | 触发出队 | `Wait`：`used ≥ 字节阈值` OR `超时(timeo参数)` OR `Close`；`Flush` 主动唤醒 ✓ |
| **D4** | 消费方式 | **用户自建消费线程**，调用 `Wait`(等触发) + `GetData`(出队)（**库不创建线程**）✓ |
| **D5** | 消费线程归属 | **由用户创建**（线程数 / 优先级 / 消费方式完全由用户决定）✓ |
| **D6** | 库名 / 前缀 | **StreamBuffer**（`StreamBufferAPI_*`，头文件 `StreamBuffer.h`）✓ |

## 六、典型应用

- **日志写文件**：用户消费线程 `GetData` 后 `fwrite` 到自己 `fopen` 的 `FILE*`（CSV 等）。
- **串口/网络攒发**：用户消费线程 `GetData` 后 `write`/`send` 一批字节，减少系统调用次数。
- **任何"攒字节流 + 定量/定时批量处理"场景**。
