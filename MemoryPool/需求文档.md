# 需求文档 — MemoryPool 内存池（固定大小对象池，配合队列减少 malloc）

> **定位**：固定大小对象的内存池 / 对象池。预分配 N 个槽位，提供 Alloc/Free（类似 malloc/free），槽位循环复用。**支持三种池满策略**（用完丢/动态扩容/阻塞等待），通过 Init 配默认模式 + 三个 Alloc 接口显式选择，适应不同场景。主要配合 ThreadQueue 消除每条消息的 malloc/free。
> **平台**：IMX6ULL (ARM Linux) | **作者**：zlzksrl | **草案**：2026-07-10
> **状态**：需求完善中（接口草案见第六节）

---

## 一、原始需求（用户提供）

1. ThreadQueue 用法要求 `PutMsg` 前申请空间、`GetMsg` 后用完 `free`。
2. 入队空间可用 `union` 固定一个长度，便于写代码。
3. 新增模块：开始 `malloc` 一片 `sizeof(struct)*N` 的空间；提供类似 `malloc` 的操作取一个空间首地址入队，`GetMsg` 后类似 `free` 归还，**循环利用**。
4. 目的：配合队列使用，减少 malloc 消耗。
5. 扩展其他使用。

> 经讨论明确：池满策略要**同时支持三种**（用完丢 / 动态扩容 / 阻塞等待），由 Init 配默认模式或调用不同 Alloc 接口选择，以适应不同场景。

## 二、需求理解：固定大小对象内存池（三模式）

### 2.1 现状痛点（ThreadQueue 典型用法）
```
生产者: p = malloc(sizeof(Msg)); 填数据; PutMsg(q,p)   ← 每条 malloc
消费者: p = GetMsg(q);          用数据; free(p)         ← 每条 free
```
高频消息 → 频繁 malloc/free → 开销大 + 内存碎片。

### 2.2 内存池方案（循环复用 + 三种满策略）
```
Init : 一次 malloc N 个固定大小槽位
生产者: p = Pool_Alloc();  填数据; PutMsg(q,p)   ← 从池取
消费者: p = GetMsg(q);     用数据; Pool_Free(p)  ← 归还池
池   : 槽位循环复用；池满时按所选策略处理
```

### 2.3 union 固定大小
所有消息类型放入一个 `union`（取最大成员长度），池槽位 = `sizeof(union)`，统一管理；池不关心内容（纯字节槽位）。

### 2.4 通用性：适合任何数据结构（与 malloc 的区别）

- 槽位是 **`element_size` 字节的纯字节块**，库**不解释内容**（与 StreamBuffer 字节流同理）——适合任意 `struct`/`union`、二进制流、文本，**含 `0x00` 也安全**（数据路径全 `memcpy`，不用字符串函数）。
- **与 malloc 的唯一区别**：每次 `Alloc()` 返回**固定 `element_size` 大小**的空间（无需传 size），`Free()` 也无需 size（固定大小）。换来：① 无运行时 malloc/碎片（槽位循环复用）；② 免 size 参数；③ O(1) Alloc/Free（空闲链表）。
- `element_size` 在 Init 时定（如 `sizeof(你的 union/struct)`），整个池统一。风格/构建与 StreamBuffer/WindowQueue 对齐（opaque pointer、`extern "C"`、Doxygen、错误码、Makefile）。

### 2.5 空间流转示意图（内嵌 next 空闲链表）

槽位在 **空闲链表（free_list）** 与 **用户** 之间循环流转，零额外管理内存：

```
==== Init: 一次 malloc N 个槽位，串成空闲链表 ====

   chunk 内存 ( N × align_size )
   ┌────────┬────────┬────────┬────────┬─────┐
   │ slot0  │ slot1  │ slot2  │ slot3  │ ... │
   └────────┴────────┴────────┴────────┴─────┘
   free_list → slot0 → slot1 → slot2 → slot3 → ... → NULL
               ↑ 每个空闲槽开头存 next(指向下一个空闲槽)

==== Alloc(): 取链头槽位给用户 ====

   free_list → slot1 → slot2 → ...          (slot0 移出链表)

   p = MemPoolAPI_Alloc();   → 返回 slot0 地址
   ┌────────┐
   │ slot0  │ → 用户拿到 align_size 字节，填数据(覆盖原 next)
   └────────┘     填完 → ThreadQueueAPI_PutMsg(q, p)  入队

==== Free(p): 用户归还，挂回链头 ====

   free_list → slot0 → slot1 → slot2 → ...
               ↑ 归还槽开头写 next=原链头(slot1)，成为新链头

   slot0: [ next=slot1 ][ 旧数据(下次 Alloc 被用户覆盖) ]

==== 循环流转 ====

        ┌─────────────────────────────────────────────┐
        ▼                                             │
   [free_list] ────Alloc()────→ [ 用户使用 ] ──Free()─┘
    空闲链表                       (align_size 字节)

==== 池满 (free_list 为空) 时，按所选模式处理 ====
   DROP  : Alloc 返回 NULL（用户自行处理）
   GROW  : malloc 新 chunk，切槽串入 free_list，继续 Alloc
   BLOCK : cond 阻塞等待，直到某线程 Free 归还槽位被唤醒
```

> **要点**：① 空闲槽位自己当链表节点（开头存 next），分配后整块给用户，**零额外内存**；② Alloc/Free 均 **O(1)**（取/挂链头）；③ GROW 扩容的 chunk 也串入同一 free_list，Free 不区分槽位来自哪个 chunk。

### 2.6 空闲链表方向与并发设计（LIFO + mutex）

**链表方向：头取头插（LIFO）** —— Alloc 取链头、Free 也挂链头。对比"头取尾插（FIFO，Free 挂链尾需 tail 指针）"：

| | LIFO（头取头插，采用） | FIFO（头取尾插） |
|---|---|---|
| 指针 | 只需 `head` | `head` + `tail` |
| 缓存 | 重用刚释放槽，CPU cache 命中高 | 轮转冷槽，cache 稍差 |
| 复杂度 | 最简 | 多 tail + 空链边界同步 |
| 轮转顺序 | LIFO（热槽优先重用） | FIFO（均匀轮转） |

FIFO"均匀轮转"对内存池无意义（池不关心内容）；LIFO 简单 + 缓存热，是 Linux slab、glibc 对象池的标准做法。**采用 LIFO。**

**并发：mutex 互斥** —— free_list 是共享资源，多线程 Alloc/Free 并发会竞态（取重、head 丢失），必须 mutex 保护。Alloc/Free 均 O(1) 几条指令、持锁极短，mutex 竞争可忽略（尤其单核 IMX6ULL）。

> 头取尾插"分离 head/tail"在有锁下**不减少竞争**（空链表/取末槽时 head 与 tail 互相关联，仍需同一把锁）。真要减竞争得用无锁结构（lock-free CAS 栈/队列），但 C 手写易错（ABA、内存序）、收益微小，**不上无锁，mutex 足够**。BLOCK 模式用 mutex + cond（Free 时 signal 唤醒等待者）。

## 三、三种池满策略（核心：适应不同场景）

| 策略 | 池满时行为 | 适用场景 | 代价 |
|------|-----------|---------|------|
| **DROP（用完丢）** | 立即返回 `NULL`，生产者自行处理（丢弃/重试） | 可丢数据、生产者不能阻塞（如日志/采集满则丢） | 可能丢数据 |
| **GROW（动态扩容）** | 再 malloc 一片新槽位加入池，返回新槽位 | 数据不能丢、内存可增长（峰值不固定） | 运行时仍有 malloc（但比每条 malloc 少） |
| **BLOCK（阻塞等待）** | 阻塞等消费者 Free 归还后再返回（可带超时） | 数据不能丢、生产者可阻塞（如指令/控制） | 生产者被卡 |

**提供方式（两者皆有，最灵活）**：
- Init 时配**默认模式** → `MemPoolAPI_Alloc()` 按默认模式。
- 三个**显式接口** → `AllocDrop()` / `AllocGrow()` / `AllocBlock(timeo)` 按指定模式（覆盖默认，同一池不同调用可用不同策略）。

## 四、功能规格

### 4.1 Init（预分配 + 默认模式）
- 入参：`element_size`、`init_count`、`mode`（默认 DROP/GROW/BLOCK）、`grow_count`（GROW 扩容粒度，**mode=GROW 时必须 >0，否则 Init 失败**）、`block_timeo`（BLOCK 默认阻塞超时 ms，0=无限）、`name`。
- 一次 malloc `element_size * init_count`，分 N 槽位，建空闲链表。
- 校验：element_size>0、init_count>0；mode=GROW 时 grow_count>0。

### 4.2 Alloc（取空闲槽位）
- 默认 `Alloc()`：按 cfg.mode。
- `AllocDrop()`：满返回 NULL。
- `AllocGrow()`：满则 malloc 新 chunk（grow_count 槽位）加入空闲链表，返回新槽位。
- `AllocBlock(timeo)`：满则 cond 阻塞等 Free（timeo=0 无限等，>0 超时返回 NULL）。
- 返回地址已按最大对齐补齐。

### 4.3 Free（归还）
- 统一 `Free(elem)`：归还槽位到空闲链表，唤醒 BLOCK 等待者（cond signal）。
- ⚠️ elem 必须是本池 Alloc 出来的（否则 UB）。GROW 扩容的槽位也归还到同一空闲链表。

### 4.4 循环利用
- Alloc/Free 复用槽位；GROW 扩容的 chunk 在 Destroy 时统一释放。

### 4.5 配合队列用法
```c
union Msg { TypeA a; TypeB b; TypeC c; };
T_MemPoolConfig cfg = { sizeof(union Msg), 64, MEMPOOL_MODE_BLOCK, 0 };
T_MemPool *pool;  MemPoolAPI_Init(&pool, &cfg, "msgpool");

union Msg *m = (union Msg *)MemPoolAPI_Alloc(pool);   /* 默认 BLOCK：满则等 */
m->a = ...;  ThreadQueueAPI_PutMsg(q, m);
/* 消费端 */ union Msg *m = GetMsg(q); ...; MemPoolAPI_Free(pool, m);

/* 或显式：某条用 DROP（可丢），某条用 GROW（必得） */
MemPoolAPI_AllocDrop(pool);
MemPoolAPI_AllocGrow(pool);
```

### 4.6 查询/统计
- 当前空闲数、已分配数、总容量（含 GROW 扩容）、峰值使用、累计 Alloc/Free 次数。

## 五、实现要点（建议）

- **空闲链表**：每个空闲槽位内嵌 `next` 指针（空闲时作节点、分配后作用户数据），O(1) Alloc/Free。
- **GROW 多 chunk**：扩容的 chunk 用链表串起，Destroy 时全部 free；空闲链表跨所有 chunk。
- **BLOCK 同步**：mutex + cond；Free 时 signal 唤醒一个等待者；AllocBlock 用 `cond_timedwait`（CLOCK_MONOTONIC）。
- **对齐**：槽位大小按 `max_align_t` 向上补齐。

## 七、公开 API 草案（待据此编写头文件）

```c
typedef struct T_MEMPOOL T_MemPool;

typedef enum {
    MEMPOOL_MODE_DROP  = 0,   /* 满则返回 NULL */
    MEMPOOL_MODE_GROW  = 1,   /* 满则动态扩容 */
    MEMPOOL_MODE_BLOCK = 2    /* 满则阻塞等待 */
} MemPoolMode;

typedef struct {
    int         element_size;  /* 槽位字节数，如 sizeof(union Msg) */
    int         init_count;    /* 初始槽位数 */
    MemPoolMode mode;          /* Alloc() 默认模式 */
    int         grow_count;    /* GROW 每次扩容新增槽位数；mode=GROW 时必须 >0，否则 Init 失败 */
    int         block_timeo;   /* BLOCK 模式默认阻塞超时(ms)，Alloc() 使用；0=无限等待，>0 超时返 NULL */
} T_MemPoolConfig;

typedef struct {
    unsigned long ulTotalAlloc;   /* 累计 Alloc 次数 */
    unsigned long ulTotalFree;    /* 累计 Free 次数 */
    unsigned long ulTotalDrop;    /* 累计 DROP 丢弃次数 */
    unsigned long ulTotalGrow;    /* 累计 GROW 扩容槽位数 */
    int           iPeakUsed;      /* 峰值已用 */
    int           iCapacity;      /* 总容量（含扩容） */
} T_MemPoolStats;

int   MemPoolAPI_Init(T_MemPool **pp, const T_MemPoolConfig *cfg, const char *name);
int   MemPoolAPI_Destroy(T_MemPool **pp);

void* MemPoolAPI_Alloc(T_MemPool *p);   /* 默认(cfg.mode)：DROP→NULL / GROW→扩容 / BLOCK→用 cfg.block_timeo 阻塞 */
void* MemPoolAPI_AllocDrop(T_MemPool *p);               /* 满则 NULL */
void* MemPoolAPI_AllocGrow(T_MemPool *p);               /* 满则扩容 */
void* MemPoolAPI_AllocBlock(T_MemPool *p, int timeo);   /* 满则阻塞；timeo=0 无限，>0 超时返 NULL */

int   MemPoolAPI_Free(T_MemPool *p, void *elem);        /* 归还（唤醒 BLOCK 等待者） */

int   MemPoolAPI_GetFreeCount(T_MemPool *p);
int   MemPoolAPI_GetUsedCount(T_MemPool *p);
int   MemPoolAPI_StatsGet(T_MemPool *p, T_MemPoolStats *st);
```

## 八、与现有库的关系

- **ThreadQueue**：主要配合对象。用池替代每条消息的 malloc/free。
- **StreamBuffer**：用户层若要把结构体 buf 从池取也可（非主要）。
- **ThreadManage_Pool**：是线程池（ThreadPool）内部实现，非通用内存池，本模块独立新建。
