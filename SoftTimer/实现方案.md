## 一、总览

### 1.1 组件清单
| 组件 | 位置 | 说明 |
|------|------|------|
| `T_SoftTimerMgr`      | 内部（Main.h） | 管理器：持有堆、live_set、锁、线程池、调度线程 |
| `T_SoftTimer`         | 内部（Main.h） | 单个定时器节点：deadline、cb、状态字段 |
| `T_SoftTimerHandle`   | 内部（Main.h） | 用户句柄：ullId、node 回指、live_set 链接 |
| min-heap              | 内部（Main.h + .c） | 数组实现，按 deadline 排序 |
| live_set              | 内部（.c）    | handle 单向链表（表头挂 mgr 上） |
| 调度线程              | 内部（.c）    | 1 个，`ThreadAPI_ThreadCreate` 创建；`pthread_cond_timedwait(CLOCK_MONOTONIC)` 睡眠 |
| 线程池                | ThreadManage 库 | `ThreadAPI_ThreadPoolCreate` 创建；`ThreadAPI_ThreadPoolAddTaskTry` 投递 cb；Destroy 时 `ThreadAPI_ThreadPoolDestroy` |

### 1.2 核心不变量（Invariants）
在任何锁外的可观察时刻：
- `INV1`：每个 `T_SoftTimer` 节点**要么**在堆中（`heap_idx >= 0`）**要么**已投递到线程池（`heap_idx < 0`），**互斥**。
- `INV2`：`node->handle != NULL` 时，`node->handle->node == node`（双向一致）。
- `INV3`：每个 handle **恰好**出现在 `live_set` 中一次（未被 Delete）或不出现（已被 Delete）。
- `INV4`：`running_count` 只能是 0 或 1；`running_tid` 仅在 `running_count==1` 时有效。
- `INV5`：`ullId` 单调递增，永不复用。

所有状态变更必须持锁 `mgr->mux` 完成。

## 三、内部数据结构完整定义

### 3.1 T_SOFTTIMERMGR

```c
struct T_SOFTTIMERMGR
{
    /* ---- 堆（按 deadline 升序，最小堆） ---- */
    T_SoftTimer     **heap;         /**< 指针数组，每格指向一个 T_SoftTimer；索引 0 是堆顶 */
    int               heap_size;    /**< 当前元素数 */
    int               heap_capacity;/**< 数组容量，动态倍增（初值 16） */

    /* ---- live_set：所有活跃/墓碑 handle 的单向链表 ---- */
    T_SoftTimerHandle *live_head;   /**< 链表表头；节点通过 handle->next_in_set 链接 */
    int                live_count;  /**< 链表节点数（含墓碑），仅用于统计与 Destroy 扫尾 */

    /* ---- ID 计数 ---- */
    unsigned long long next_id;     /**< 下一个 handle 的 ullId，单调递增，不复用 */

    /* ---- 调度线程 ---- */
    pthread_t         sched_tid;    /**< 调度线程 PID */
    int               stop_flag;    /**< 1 = 请求调度线程退出（Destroy 阶段） */

    /* ---- 同步原语 ---- */
    pthread_mutex_t   mux;          /**< 单一大锁，保护本结构体所有字段 */
    pthread_cond_t    cv_sched;     /**< 调度线程等待：堆顶变更 / 新定时器 / 停止 */
    pthread_cond_t    cv_delete;    /**< Delete 等待：running_count 变 0 */

    /* ---- 线程池（ThreadManage） ---- */
    T_ThreadPoolHandle *pool;

    /* ---- 运行统计 ---- */
    unsigned long ulTotalSet;
    unsigned long ulTotalFired;
    unsigned long ulTotalCanceled;
    unsigned long ulTotalOverrun;
    int           iActiveCount;
    int           iPeakActive;

    /* ---- 元数据 ---- */
    char name[32];                  /**< 管理器名，Init 时从 t_Config.sMgrName 拷贝 */
    int  init_done;                 /**< 1 = Init 完成，Destroy 前有效 */
};
```

### 3.2 T_SOFTTIMER

```c
/* 内部状态枚举 —— 与 heap_idx/running_count 配合定义节点位置 */
enum E_SoftTimerState
{
    ST_IN_HEAP    = 0,   /**< 在堆中等待触发 */
    ST_DISPATCHED = 1,   /**< 已投递到线程池，wrapper 未开始执行 */
    ST_RUNNING    = 2,   /**< wrapper 正在执行 user_cb */
    ST_ORPHANED   = 3,   /**< 已从 heap/pool 脱离且 Delete 已取走 handle；wrapper 收尾即释放 */
};

struct T_SOFTTIMER
{
    /* ---- 反向指针 ---- */
    T_SoftTimerHandle *handle;      /**< 回指句柄；Delete 分离后置 NULL */

    /* ---- 计时 ---- */
    struct timespec deadline;       /**< 下一次触发的绝对时刻（CLOCK_MONOTONIC） */
    struct timespec last_scheduled; /**< 上一次预定触发时刻（供 FROM_SCHEDULED 计算） */
    int             iPeriodMs;      /**< 周期，0 = 单次 */
    E_SoftTimerPeriodMode ePeriodMode;

    /* ---- 用户回调 ---- */
    SoftTimerCb     cb;
    void           *user_ctx;

    /* ---- 状态（均在 mgr->mux 内改） ---- */
    enum E_SoftTimerState state;    /**< 见 §五 状态机 */
    int                   is_canceled;/**< 1 = Delete 已取消，wrapper 收尾时跳过续期 */
    int                   running_count; /**< 0 或 1；ST_RUNNING 时为 1 */
    pthread_t             running_tid;   /**< 仅 running_count==1 时有效 */
    int                   heap_idx;      /**< 在 mgr->heap 中的索引，-1 = 不在堆中 */

    /* ---- 元数据 ---- */
    char timer_name[32];
};
```

### 3.3 T_SOFTTIMERHANDLE

```c
struct T_SOFTTIMERHANDLE
{
    unsigned long long ullId;               /**< 单调递增序号，仅日志/stats 用 */
    T_SoftTimer       *node;                /**< 内部节点回指；单次 fire 后成墓碑时为 NULL */
    int                is_consumed;         /**< 1 = 单次已 fire 且 node 已释放（墓碑） */
    T_SoftTimerHandle *next_in_set;         /**< live_set 链表下一节点 */
};
```

### 3.4 关系图

```
    T_SoftTimerMgr
    ┌─────────────────┐
    │ heap[] ─────────┼──▶ [T_SoftTimer*]  [T_SoftTimer*]  ...
    │                 │           │                │
    │                 │           ▼                ▼
    │ live_head ──────┼──▶ [T_SoftTimerHandle]  [T_SoftTimerHandle]  ...
    │                 │        │  next_in_set ──▶ next_in_set ──▶ ...
    │                 │        │
    │ mux/cv_sched/   │        │
    │ cv_delete       │        ▼
    │ pool ───────────┼──▶ ThreadPool     ┌──── node ────┐
    │ sched_tid       │                   ▼              │
    └─────────────────┘             T_SoftTimer ─── handle
                                                (双向)
```
- **每个 `T_SoftTimer`** 通过 `handle` 反指到 `T_SoftTimerHandle`；
- **每个 `T_SoftTimerHandle`** 通过 `node` 指向 `T_SoftTimer`（墓碑时为 NULL）；
- 堆按 deadline 排序；live_set 用于 Delete 时的**指针身份查询**（O(N) 遍历，N 小时可接受）。

## 四、支撑数据结构

### 4.1 min-heap（数组实现）

**为何选数组堆而非红黑树/时间轮**：
- 定时器数量典型 N < 100，`O(log N)` 与 `O(1)` 差异微小；
- 无外部依赖（红黑树需要额外文件）；
- 内存局部性好（数组连续，缓存友好）；
- 实现简单：sift-up / sift-down 共约 40 行代码。

**操作接口**（内部 `static` 函数）：
| 函数 | 复杂度 | 说明 |
|------|--------|------|
| `heap_push(mgr, node)` | O(log N) | 尾插入 + sift-up；同步更新 `node->heap_idx` |
| `heap_pop_top(mgr)`    | O(log N) | 取堆顶 + 末尾交换 + sift-down；返回节点 |
| `heap_remove_at(mgr, idx)` | O(log N) | 删任意位置：末尾交换后 sift-down + sift-up；用于 Delete 从堆中拔节点 |
| `heap_peek_top(mgr)`   | O(1)     | 只读堆顶（scheduler 用于计算睡眠时间） |
| `heap_ensure_capacity(mgr, n)` | 均摊 O(1) | 容量不足时倍增 realloc |

**关键实现细节**：
- 交换两节点时，**同步更新它们的 `heap_idx` 字段** —— 这是 `heap_remove_at` 能 O(log N) 的关键（否则要 O(N) 线性搜索）。
- 比较键：`timespec_cmp(&a->deadline, &b->deadline)`（先比 `tv_sec` 再比 `tv_nsec`）。

伪码片段：
```c
static void heap_swap(T_SoftTimerMgr *mgr, int i, int j) {
    T_SoftTimer *ni = mgr->heap[i], *nj = mgr->heap[j];
    mgr->heap[i] = nj;  nj->heap_idx = i;
    mgr->heap[j] = ni;  ni->heap_idx = j;
}
```

### 4.2 live_set（handle 单向链表）

**为何用单向链表而非哈希表**：
- N 小（典型 < 100），O(N) 查找可接受（Delete 不是热点路径）；
- 无需 hash 函数、无桶数组、无 rehash；
- 表头插入 O(1)；链接指针内嵌在 handle 自身（`next_in_set` 字段），零额外分配。

**操作接口**（内部 `static`）：
| 函数 | 复杂度 | 说明 |
|------|--------|------|
| `liveset_add(mgr, h)` | O(1) | 表头插入 |
| `liveset_find(mgr, ptr)` | O(N) | 按指针地址遍历比较，**不解引用 `ptr`** |
| `liveset_remove(mgr, h)` | O(N) | 单向链表移除（需前驱指针） |

**关键**：`liveset_find` **只比较指针值**、**从不解引用用户传入的 `ptr`**。用户传野指针也不会 crash。

伪码：
```c
static T_SoftTimerHandle *liveset_find(T_SoftTimerMgr *mgr,
                                       T_SoftTimerHandle *user_ptr)
{
    for (T_SoftTimerHandle *h = mgr->live_head; h; h = h->next_in_set)
        if (h == user_ptr) return h;   /* 指针值相等，不 deref user_ptr */
    return NULL;
}
```

### 4.3 单调 ID 计数器
- `mgr->next_id` 初值 1（0 保留为无效）。
- 每次 `SetAlarm` 成功分配后 `handle->ullId = mgr->next_id++`。
- 溢出：`unsigned long long` 每纳秒递增也需 500+ 年才溢出，实际不必处理。

## 五、状态机与所有权

### 5.1 T_SoftTimer 状态转移

```
                  ┌──────────┐
   SetAlarm ─────▶│ IN_HEAP  │◀─── wrapper 收尾（周期，未取消）
                  └────┬─────┘
                       │ 调度线程 pop + ThreadPoolAddTask
                       ▼
                  ┌──────────┐
                  │DISPATCHED│
                  └────┬─────┘
                       │ wrapper 开始执行（!is_canceled）
                       ▼
                  ┌──────────┐
                  │ RUNNING  │──── running_count=1, running_tid=self
                  └────┬─────┘
       ┌───────────────┼───────────────┐
       │               │               │
   Delete(外线程)   is_canceled     周期正常
   set canceled=1     wrapper cb 完成  wrapper 续期
       ▼               │               ▼
   等 cv_delete ◀──────┴──────▶  回 IN_HEAP
       │
       ▼
   node 由 wrapper 或
   本 Delete 释放（见 5.2）

   任何非 RUNNING 状态下 Delete → 直接摘除并释放节点
   自删除路径（RUNNING 且 tid==self）→ 状态转为 ORPHANED，wrapper 收尾释放
```

### 5.2 内存所有权规则

**核心原则**：**同一时刻节点只有一个"责任释放方"**。规则：

| 场景 | 状态 | 谁 free(node) | 谁 free(handle) |
|------|------|--------------|-----------------|
| SetAlarm 成功 | IN_HEAP | — | — |
| Delete 未运行（node 在 heap） | IN_HEAP | **Delete** | **Delete** |
| Delete 已投递未运行 | DISPATCHED | **wrapper 早退分支** | **Delete**（先摘除 handle） |
| Delete 运行中（他线程） | RUNNING → 等待 → 收尾 | **wrapper 收尾** | **Delete**（等 cv 后释放） |
| Delete 运行中（自删除） | RUNNING（self） | **wrapper 收尾** | **Delete**（立即释放） |
| 单次 cb 正常完成 | RUNNING → 收尾 | **wrapper**（同时给 handle 打墓碑） | 后续用户 Delete 或 mgr Destroy 扫尾 |
| Destroy 扫尾 | any | **Destroy** | **Destroy** |

**关键约定**：
- 一旦 Delete 决定"由 wrapper 释放 node"（DISPATCHED/RUNNING 场景），Delete 必须**先设 `is_canceled=1` 并将 `node->handle = NULL`**，确保 wrapper 收尾时不会碰已释放的 handle。
- wrapper 通过 `node->handle == NULL` 与 `node->is_canceled` 两个信号，决定收尾时是否触碰 handle。

## 六、并发原语

### 6.1 锁划分（单锁策略）

- **`mgr->mux`（一把大锁）**：保护堆、live_set、所有节点/句柄字段、统计字段、`stop_flag`。
- **不做锁分离**（如 heap 锁 + live_set 锁）：
  - 复杂度低；
  - 定时器操作频率不高（心跳/控制环量级），单锁性能够；
  - Delete 涉及跨结构（heap + live_set + node），分离锁会引入锁序问题。

### 6.2 条件变量

**`cv_sched`**：调度线程用。
- 调度线程 `pthread_cond_timedwait(cv_sched, mux, &heap_top_deadline)` 睡眠。
- 唤醒者：
  - `SetAlarm` 插入了比堆顶更早的 deadline → 调 `pthread_cond_signal(cv_sched)`。
  - `Delete` 删掉了堆顶 → 同上。
  - `Destroy` 设置 `stop_flag=1` 后 `pthread_cond_broadcast(cv_sched)`。

**`cv_delete`**：外线程 Delete 用。
- Delete 在 `RUNNING` 场景下 `pthread_cond_wait(cv_delete, mux)` 等待 `running_count==0`。
- 唤醒者：wrapper 在 cb 完成、`running_count = 0` 后 `pthread_cond_broadcast(cv_delete)`。
- **用 broadcast 而非 signal**：允许多个不同 Delete 各自等待自己的 node（用统一 cv 简化实现，谓词各自判定）。

**cond var 时钟绑定**（**关键**）：
```c
pthread_condattr_t attr;
pthread_condattr_init(&attr);
pthread_condattr_setclock(&attr, CLOCK_MONOTONIC);  /* ← 必须，否则 timedwait 用 REALTIME */
pthread_cond_init(&mgr->cv_sched, &attr);
pthread_cond_init(&mgr->cv_delete, &attr);
pthread_condattr_destroy(&attr);
```

### 6.3 死锁避免与自删除

**唯一死锁风险**：回调内调 `Delete(自己)` 且 Delete 等待 `cv_delete` → 自己等自己完成 → 死锁。

**规避**：Delete 进入等待前，**先检查 `pthread_equal(pthread_self(), node->running_tid)`**：
- 相等 → 自删除路径：**不等**，标记 `is_canceled=1` + `node->handle=NULL`，摘 live_set，释放 handle，`*ppt=NULL`，返回 0。node 由 wrapper 收尾释放。
- 不等 → 外线程路径：`pthread_cond_wait(cv_delete)` 等 wrapper 完成。

**回调内调 `SetAlarm` 是安全的**：SetAlarm 只加 mgr->mux（可重入吗？—— **不可重入**）；因此回调内**必须在锁外**才能调 SetAlarm。wrapper 在调 user_cb 之前 `unlock(mux)`，回调期无锁，SetAlarm 可以重新加锁。同理回调内调 Delete。

**回调内调 mgr 已销毁的 API**：由用户负责不做（Destroy 是"最后一步"契约）。

## 七、关键流程实现

### 7.1 Init

```c
int SoftTimerAPI_Init(T_SoftTimerMgr **ppt_Mgr, const T_SoftTimerConfig t_Config)
{
    if (!ppt_Mgr || *ppt_Mgr != NULL) return -1;
    if (t_Config.iMinNum <= 0 || t_Config.iMaxNum < t_Config.iMinNum
        || t_Config.iQueueMaxSize <= 0) return -1;

    T_SoftTimerMgr *mgr = calloc(1, sizeof(*mgr));
    if (!mgr) return -1;

    strncpy(mgr->name, t_Config.sMgrName, sizeof(mgr->name)-1);
    mgr->name[sizeof(mgr->name)-1] = '\0';

    mgr->heap_capacity = 16;
    mgr->heap = calloc(mgr->heap_capacity, sizeof(T_SoftTimer *));
    if (!mgr->heap) { free(mgr); return -1; }

    mgr->next_id = 1;

    /* 互斥锁 */
    pthread_mutex_init(&mgr->mux, NULL);

    /* 条件变量（绑 CLOCK_MONOTONIC） */
    pthread_condattr_t attr;
    pthread_condattr_init(&attr);
    pthread_condattr_setclock(&attr, CLOCK_MONOTONIC);
    pthread_cond_init(&mgr->cv_sched,  &attr);
    pthread_cond_init(&mgr->cv_delete, &attr);
    pthread_condattr_destroy(&attr);

    /* 线程池 */
    T_ThreadPoolConfig pool_cfg = {
        .iMinNum        = t_Config.iMinNum,
        .iMaxNum        = t_Config.iMaxNum,
        .iQueueMaxSize  = t_Config.iQueueMaxSize,
        .iIdleTimeoutMs = t_Config.iIdleTimeoutMs,
    };
    mgr->pool = ThreadAPI_ThreadPoolCreate(pool_cfg);
    if (!mgr->pool) goto fail_pool;

    /* 调度线程 */
    T_ThreadCreateConfig th_cfg = {
        .pThreadFunc         = softtimer_sched_thread_main,
        .pThreadFuncUserArg  = mgr,
        .sThreadName         = mgr->name,   /* 用作调试名前缀 */
        .eSetAttr            = (t_Config.iSchedPriority > 0) ? 2 : 0,
        .eSchedPolicy        = SCHED_FIFO,
        .iSchedPriority      = t_Config.iSchedPriority,
        .einheritsched       = PTHREAD_EXPLICIT_SCHED,
        .eDetachState        = PTHREAD_CREATE_JOINABLE,
    };
    if (ThreadAPI_ThreadCreate(&th_cfg) < 0) goto fail_sched;
    mgr->sched_tid = th_cfg.tThreadPid;

    mgr->init_done = 1;
    *ppt_Mgr = mgr;
    return 0;

fail_sched:
    ThreadAPI_ThreadPoolDestroy(mgr->pool);
fail_pool:
    pthread_cond_destroy(&mgr->cv_sched);
    pthread_cond_destroy(&mgr->cv_delete);
    pthread_mutex_destroy(&mgr->mux);
    free(mgr->heap);
    free(mgr);
    return -2;
}
```

### 7.2 Destroy

```c
int SoftTimerAPI_Destroy(T_SoftTimerMgr **ppt_Mgr)
{
    if (!ppt_Mgr || !*ppt_Mgr) return -1;
    T_SoftTimerMgr *mgr = *ppt_Mgr;

    /* 1. 通知调度线程退出 */
    pthread_mutex_lock(&mgr->mux);
    mgr->stop_flag = 1;
    pthread_cond_broadcast(&mgr->cv_sched);
    pthread_mutex_unlock(&mgr->mux);

    /* 2. 等调度线程结束 */
    pthread_join(mgr->sched_tid, NULL);

    /* 3. 销毁线程池（会等所有 in-flight cb 完成） */
    ThreadAPI_ThreadPoolDestroy(mgr->pool);

    /* 4. 扫尾所有残留节点与句柄（此时无并发） */
    for (int i = 0; i < mgr->heap_size; i++) {
        T_SoftTimer *n = mgr->heap[i];
        if (n->handle) n->handle->node = NULL;
        free(n);
    }
    while (mgr->live_head) {
        T_SoftTimerHandle *h = mgr->live_head;
        mgr->live_head = h->next_in_set;
        free(h);
    }

    /* 5. 释放资源 */
    pthread_cond_destroy(&mgr->cv_sched);
    pthread_cond_destroy(&mgr->cv_delete);
    pthread_mutex_destroy(&mgr->mux);
    free(mgr->heap);
    free(mgr);
    *ppt_Mgr = NULL;
    return 0;
}
```

### 7.3 SetAlarm

```c
int SoftTimerAPI_SetAlarm(T_SoftTimerMgr *mgr,
                          const T_SoftTimerAlarm t_Alarm,
                          T_SoftTimerHandle **ppt_Handle)
{
    if (!mgr || !ppt_Handle || *ppt_Handle) return -1;
    if (!t_Alarm.cb || t_Alarm.iFirstDelayMs < 0 || t_Alarm.iPeriodMs < 0) return -1;

    T_SoftTimer       *node = calloc(1, sizeof(*node));
    T_SoftTimerHandle *h    = calloc(1, sizeof(*h));
    if (!node || !h) { free(node); free(h); return -1; }

    struct timespec now;
    clock_gettime(CLOCK_MONOTONIC, &now);

    node->handle       = h;
    node->deadline     = timespec_add_ms(now, t_Alarm.iFirstDelayMs);
    node->last_scheduled = node->deadline;
    node->iPeriodMs    = t_Alarm.iPeriodMs;
    node->ePeriodMode  = t_Alarm.ePeriodMode;
    node->cb           = t_Alarm.cb;
    node->user_ctx     = t_Alarm.user_ctx;
    node->state        = ST_IN_HEAP;
    node->heap_idx     = -1;
    strncpy(node->timer_name, t_Alarm.sTimerName, sizeof(node->timer_name)-1);

    pthread_mutex_lock(&mgr->mux);
    if (mgr->stop_flag) { pthread_mutex_unlock(&mgr->mux); free(node); free(h); return -2; }

    h->ullId       = mgr->next_id++;
    h->node        = node;
    h->is_consumed = 0;
    liveset_add(mgr, h);

    int old_top = (mgr->heap_size > 0) ? mgr->heap[0]->heap_idx : -1;
    heap_push(mgr, node);

    /* 若新节点成为堆顶，唤醒调度线程重算睡眠 */
    if (mgr->heap[0] == node) pthread_cond_signal(&mgr->cv_sched);

    mgr->ulTotalSet++;
    mgr->iActiveCount++;
    if (mgr->iActiveCount > mgr->iPeakActive) mgr->iPeakActive = mgr->iActiveCount;

    *ppt_Handle = h;
    pthread_mutex_unlock(&mgr->mux);
    return 0;
}
```

### 7.4 Delete —— 三条路径

```c
int SoftTimerAPI_Delete(T_SoftTimerMgr *mgr, T_SoftTimerHandle **ppt_Handle)
{
    if (!mgr || !ppt_Handle || !*ppt_Handle) return -1;
    T_SoftTimerHandle *user_ptr = *ppt_Handle;

    pthread_mutex_lock(&mgr->mux);
    if (mgr->stop_flag) { pthread_mutex_unlock(&mgr->mux); return -2; }

    T_SoftTimerHandle *h = liveset_find(mgr, user_ptr);
    if (!h) { pthread_mutex_unlock(&mgr->mux); return -3; }

    /* -- 路径 A：墓碑（单次已 fire） -- */
    if (h->is_consumed) {
        liveset_remove(mgr, h);
        free(h);
        *ppt_Handle = NULL;
        pthread_mutex_unlock(&mgr->mux);
        return 0;
    }

    T_SoftTimer *node = h->node;   /* != NULL 因非墓碑 */

    /* -- 路径 B：未运行（在堆中） -- */
    if (node->state == ST_IN_HEAP) {
        heap_remove_at(mgr, node->heap_idx);
        liveset_remove(mgr, h);
        free(node);
        free(h);
        mgr->ulTotalCanceled++;
        mgr->iActiveCount--;
        *ppt_Handle = NULL;
        pthread_mutex_unlock(&mgr->mux);
        return 0;
    }

    /* -- 路径 C：已投递未运行（DISPATCHED） -- */
    if (node->state == ST_DISPATCHED) {
        node->is_canceled = 1;
        node->handle      = NULL;   /* wrapper 收尾时不碰 handle */
        liveset_remove(mgr, h);
        free(h);
        mgr->ulTotalCanceled++;
        mgr->iActiveCount--;
        *ppt_Handle = NULL;
        /* node 由 wrapper 早退分支释放 */
        pthread_mutex_unlock(&mgr->mux);
        return 0;
    }

    /* -- 路径 D：RUNNING -- */
    /* D1：自删除 */
    if (pthread_equal(pthread_self(), node->running_tid)) {
        node->is_canceled = 1;
        node->handle      = NULL;
        node->state       = ST_ORPHANED;
        liveset_remove(mgr, h);
        free(h);
        mgr->ulTotalCanceled++;
        mgr->iActiveCount--;
        *ppt_Handle = NULL;
        /* node 由 wrapper 收尾释放 */
        pthread_mutex_unlock(&mgr->mux);
        return 0;
    }

    /* D2：他线程等 */
    node->is_canceled = 1;
    node->handle      = NULL;       /* 先切断，wrapper 不再触碰 handle */
    liveset_remove(mgr, h);         /* 从 live_set 摘除，防并发 Delete */
    while (node->running_count > 0) {
        pthread_cond_wait(&mgr->cv_delete, &mgr->mux);
    }
    /* wrapper 已完成本轮 cb；因 is_canceled，wrapper 收尾释放了 node，或即将释放 */
    /* 但 wrapper 在锁外走完 cb 后重新加锁做收尾；由于我们持锁，wrapper 现在等我们释放 */
    /* 事实上 cv_wait 醒来后，running_count 已为 0 但 wrapper 尚未继续（因我们抢到锁） */
    /* wrapper 收尾会释放 node —— 但如果我们此刻返回并释放 mutex，wrapper 才继续 */
    /* 为清晰起见：让 wrapper 释放 node（因 node->handle=NULL, is_canceled=1）*/
    free(h);
    mgr->ulTotalCanceled++;
    mgr->iActiveCount--;
    *ppt_Handle = NULL;
    pthread_mutex_unlock(&mgr->mux);
    return 0;
}
```

> **注意 D2 的执行顺序**：wrapper 在锁外调 cb → cb 结束 → wrapper 重新加锁 → 设 `running_count=0` → `cond_broadcast(cv_delete)` → 检查 `is_canceled` → 释放 node → 解锁返回。
> Delete 的 cv_wait 只有等 wrapper 释放锁后才能真正醒来；此时 wrapper 已释放 node。所以 Delete 的 "等 running_count==0" 谓词表达的是**wrapper 本轮已收尾完成**。node 已被 wrapper 释放，Delete 不再触碰。

### 7.5 调度线程主循环

```c
static void *softtimer_sched_thread_main(void *arg)
{
    T_SoftTimerMgr *mgr = arg;

    pthread_mutex_lock(&mgr->mux);
    while (!mgr->stop_flag) {

        if (mgr->heap_size == 0) {
            pthread_cond_wait(&mgr->cv_sched, &mgr->mux);
            continue;
        }

        struct timespec now, dl = mgr->heap[0]->deadline;
        clock_gettime(CLOCK_MONOTONIC, &now);

        if (timespec_cmp(&dl, &now) > 0) {
            /* 未到点：睡到 dl 或被唤醒 */
            pthread_cond_timedwait(&mgr->cv_sched, &mgr->mux, &dl);
            continue;   /* 醒来后重看堆顶 */
        }

        /* 到点：批量摘出所有 deadline <= now 的节点，投递线程池 */
        while (mgr->heap_size > 0
               && timespec_cmp(&mgr->heap[0]->deadline, &now) <= 0)
        {
            T_SoftTimer *node = heap_pop_top(mgr);
            node->state = ST_DISPATCHED;
            node->heap_idx = -1;

            /* 投递到线程池（可能阻塞若队列满；我们持锁不好阻塞）
             * → 用 Try 版本，队列满则记 overrun 并 requeue */
            if (ThreadAPI_ThreadPoolAddTaskTry(mgr->pool,
                                               softtimer_wrapper, node) < 0)
            {
                /* 池满：节点回堆稍后重试；overrun 计数 */
                node->state = ST_IN_HEAP;
                node->deadline = timespec_add_ms(now, 1);   /* 1ms 后重试 */
                heap_push(mgr, node);
                mgr->ulTotalOverrun++;
                break;   /* 让出锁，稍后再来 */
            }
            mgr->ulTotalFired++;
        }
    }
    pthread_mutex_unlock(&mgr->mux);
    return NULL;
}
```

**说明**：
- 用 `ThreadPoolAddTaskTry`（非阻塞）而非 `AddTask`（阻塞）—— 持锁阻塞会锁死其他 API。
- 池满走 requeue + 1ms 退避 + overrun 计数，实现"背压"。
- 每轮循环重看堆顶：`SetAlarm` 插入新最早 deadline 时通过 `cond_signal` 唤醒。

### 7.6 线程池 wrapper

```c
static void *softtimer_wrapper(void *arg)
{
    T_SoftTimer *node    = arg;
    T_SoftTimerMgr *mgr  = /* 从 node 反推？或者 wrapper 参数打包结构体传 */;
    /* 实际实现：Init 时 mgr 是唯一的，wrapper 需要 mgr —— 用一个 wrapper_arg 结构：
     * struct { T_SoftTimerMgr *mgr; T_SoftTimer *node; } * */
    /* 简化：node 中带 mgr 反指字段，或用 thread-local。这里假设已可拿到 mgr。 */

    pthread_mutex_lock(&mgr->mux);

    /* 早退分支：DISPATCHED 期间被 Delete 取消了 */
    if (node->is_canceled) {
        pthread_mutex_unlock(&mgr->mux);
        free(node);   /* 见 §5.2 所有权表 DISPATCHED 行 */
        return NULL;
    }

    node->state         = ST_RUNNING;
    node->running_count = 1;
    node->running_tid   = pthread_self();
    T_SoftTimerHandle *h_snapshot = node->handle;   /* 快照，锁外传给 cb */
    SoftTimerCb  cb_snapshot  = node->cb;
    void        *ctx_snapshot = node->user_ctx;
    pthread_mutex_unlock(&mgr->mux);

    /* 调用用户回调（锁外） */
    cb_snapshot(h_snapshot, ctx_snapshot);

    pthread_mutex_lock(&mgr->mux);
    node->running_count = 0;
    pthread_cond_broadcast(&mgr->cv_delete);   /* 唤醒可能等待的 Delete */

    if (node->is_canceled) {
        /* Delete 已取消（含自删除 ORPHANED），wrapper 负责释放 node */
        pthread_mutex_unlock(&mgr->mux);
        free(node);
        return NULL;
    }

    if (node->iPeriodMs == 0) {
        /* 单次正常完成：node 释放，handle 打墓碑 */
        if (node->handle) {
            node->handle->is_consumed = 1;
            node->handle->node        = NULL;
        }
        mgr->iActiveCount--;
        pthread_mutex_unlock(&mgr->mux);
        free(node);
        return NULL;
    }

    /* 周期正常：重算 deadline，回堆 */
    struct timespec now;
    clock_gettime(CLOCK_MONOTONIC, &now);
    node->deadline = compute_next_deadline(node, &now);   /* 见 §八 */
    node->state    = ST_IN_HEAP;
    heap_push(mgr, node);
    if (mgr->heap[0] == node) pthread_cond_signal(&mgr->cv_sched);

    pthread_mutex_unlock(&mgr->mux);
    return NULL;
}
```

**wrapper 拿 mgr 指针的做法**（伪码里省略了）：定义一个内部 `T_WrapperArg { T_SoftTimerMgr *mgr; T_SoftTimer *node; }`，调度线程 dispatch 时 malloc 一份、wrapper 完成释放。或者 `T_SoftTimer` 结构体再加一个 `struct T_SOFTTIMERMGR *mgr` 反指字段（更省 malloc，推荐）。

## 八、周期漂移算法

### 8.1 FROM_END

```c
/* 回调结束时刻 now 已经在 wrapper 内取好 */
node->deadline       = timespec_add_ms(now, node->iPeriodMs);
node->last_scheduled = node->deadline;
/* overrun 判定：本次预定 vs 实际完成 */
if (timespec_diff_ms(&now, &node->last_scheduled) > node->iPeriodMs)
    mgr->ulTotalOverrun++;
```

### 8.2 FROM_SCHEDULED（含 snap）

```c
/* naive: node->last_scheduled += period */
struct timespec naive = timespec_add_ms(node->last_scheduled, node->iPeriodMs);

if (timespec_cmp(&naive, &now) > 0) {
    /* 未超期：正常栅格 */
    node->deadline       = naive;
    node->last_scheduled = naive;
} else {
    /* 超期：snap 到下一未来 grid */
    long long delta_ms  = timespec_diff_ms(&now, &node->last_scheduled);
    long long missed    = (delta_ms + node->iPeriodMs - 1) / node->iPeriodMs;  /* ceil */
    struct timespec next = timespec_add_ms(node->last_scheduled,
                                           missed * node->iPeriodMs);
    node->deadline       = next;
    node->last_scheduled = next;
    mgr->ulTotalOverrun += (missed - 1);   /* 跳过的 grid 数 */
}
```

### 8.3 overrun 计数
- FROM_END：仅当"cb 完成时刻 - 本次预定 deadline" > period 时 +1。
- FROM_SCHEDULED：`missed - 1`（若 missed=1 表示按栅格正常，不算跳过）。

## 九、并发场景 sequence 图

### 9.1 场景 A：Delete 未触发（IN_HEAP）
```
用户线程                    调度线程
   │                            │
   ├─ lock(mux)                 │ (在 cv_sched 睡眠)
   ├─ liveset_find OK           │
   ├─ heap_remove_at            │
   ├─ free node, free h         │
   ├─ *ppt = NULL               │
   ├─ unlock ────────────▶      │
                                │ (heap 变了，但不影响堆顶就无唤醒)
```

### 9.2 场景 B：Delete 时 cb 正在运行（他线程）
```
用户线程 X                 wrapper 线程                       调度线程
    │                          │                                 │
    │                       (cb 执行中，无锁)                     │
X: lock(mux)                   │                                 │
X: liveset_find OK             │                                 │
X: state == RUNNING            │                                 │
X: tid != self                 │                                 │
X: is_canceled = 1             │                                 │
X: node->handle = NULL         │                                 │
X: liveset_remove; free(h)     │                                 │
X: cond_wait(cv_delete) ─┐     │                                 │
        (释放锁，等待)   │     │                                 │
                         │ cb 完成                                │
                         │ lock(mux)                              │
                         │ running_count = 0                      │
                         │ cond_broadcast(cv_delete) ──▶ 唤醒 X   │
                         │ is_canceled? 是                        │
                         │ free(node)                             │
                         │ unlock ─────────────────────────       │
X: 重获锁; running_count==0    │                                 │
X: unlock; return 0            │                                 │
```

### 9.3 场景 C：cb 内自删除
```
wrapper 线程                                         调度线程
    │                                                    │
lock; state=RUNNING; running_tid=self; unlock            │
cb(handle, ctx):                                         │
    ┌──────────────────────────────────────────┐         │
    │ SoftTimerAPI_Delete(mgr, &copy_h):       │         │
    │   lock                                   │         │
    │   liveset_find OK                        │         │
    │   state==RUNNING, tid==self  ← 检测到自删│         │
    │   is_canceled=1; handle=NULL; state=ORPHANED│      │
    │   liveset_remove; free(h)                │         │
    │   *ppt=NULL; unlock; return 0            │         │
    └──────────────────────────────────────────┘         │
cb 返回                                                  │
lock; running_count=0; cond_broadcast(cv_delete)         │
is_canceled? 是                                          │
free(node); unlock                                       │
```

### 9.4 场景 D：单次 fire → 用户后续 Delete
```
wrapper 线程                                    用户线程
    │                                              │
lock; RUNNING; unlock; cb; lock                    │
running_count=0; broadcast                         │
is_canceled? 否                                    │
iPeriodMs == 0：                                   │
  handle->is_consumed=1; handle->node=NULL         │
  iActiveCount--                                    │
free(node); unlock                                 │
                                                   │ (稍后) Delete(&h):
                                                   │   lock
                                                   │   liveset_find OK
                                                   │   is_consumed==1  ← 墓碑
                                                   │   liveset_remove; free(h); *ppt=NULL
```

### 14.2 已知设计权衡（非缺陷）
| 权衡 | 说明 | 参考 |
|---|---|---|
| `mgr->mux` 无 PI 协议 | V1.0 默认 SCHED_OTHER；若用户开 SCHED_FIFO 且有长跑低优线程，存在优先级反转窗口 | 设计探讨.md §2.3 |
| `ST_POOL_ADDTASK_RETRY_MS = 10` 硬编码 | 池满退避下限；对 <10ms 周期不友好 | 需求文档.md §4.1 |
| FROM_END 池满 requeue 可能双计 overrun | 一次 requeue + 一次 wrapper 收尾判定；粗粒度背压指标，趋势可信 | AI审查结果.md 三轮 L16 |
| `live_set` O(N) 查询 | N 典型 < 100 时可接受；Delete 非热点 | 实现方案.md §4.2 |
| 周期定时器 heap_push 失败墓碑化 | realloc 失败时提前终止，仅 `ST_LOGE` 日志，统计字段不体现 | SoftTimer.h 文件级"极端资源不足行为" |
| worker 线程优先级未透传 | `iSchedPriority` 只影响 sched 线程；worker 仍是 SCHED_OTHER | 设计探讨.md §2.2 |

## 十五、遗留问题与后续（V1.1）

### 15.1 功能扩展
- `Adjust(handle, iNewPeriodMs)`：需要在 RUNNING 状态下也能调整；实现需再想清楚状态机。
- `Pause/Resume(handle)`：状态机加 `ST_PAUSED`；调度线程遇 paused 节点直接跳过。
- `DeleteByCallback(mgr, cb)`：遍历 live_set，批量 Delete；注意迭代器失效。

### 15.2 性能与实时性
- 换 live_set 为 hash 表（N > 1000 时收益明显）。
- 换 heap 为红黑树（若需要 O(log N) 按 handle 定位到堆位置）。
- worker 线程优先级透传（需 `ThreadManage` 先支持 `T_ThreadPoolConfig.iWorkerPriority`）。
- `mgr->mux` 加 `PTHREAD_PRIO_INHERIT`（配合上一项，用于软实时场景）。
- `ST_POOL_ADDTASK_RETRY_MS` 从常量改为 `T_SoftTimerConfig` 字段。

### 15.3 可观测性
- `ulTotalOverrun` 拆分三个独立字段（FROM_END 超时 / FROM_SCHEDULED snap / 池满 requeue），细化背压来源。
- 新增 `ulTotalDropped` 记录周期定时器 heap_push 失败墓碑化次数。
- 新增 `SoftTimerAPI_ListSnapshot(mgr, buf, cap, &n)` 遍历 live_set 供运维监控。

### 15.4 时钟精度
- 备选后端 `ST_CONFIG_USE_TIMERFD`（内部实现开关，默认关）：把 `cond_timedwait` 替换为 `epoll_wait + timerfd`，微秒级场景 POC 用。详见 `设计探讨.md §3.4`。
- `struct timespec` 入参重载：`iFirstDelayNs / iPeriodNs` 或直接 timespec 入参，支持亚毫秒规格。
