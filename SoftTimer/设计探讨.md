# SoftTimer 设计探讨

> 本文档记录 SoftTimer 二轮审查修复后的三个设计层面探讨，不属于需求或实现规格，
> 用于回答"精度到什么程度"、"优先级如何影响"、"能否用内核机制替换"这类工程问题。
> 目标平台：IMX6ULL (ARM Linux) | 记录日期：2026-07-14

---

## 一、定时器最小单位时间

从三个层面看，答案不同。

### 1.1 API 层：1 毫秒

公共 API 全部用 `int` 承载毫秒：

```c
int iFirstDelayMs;  // SoftTimer.h:111
int iPeriodMs;      // SoftTimer.h:112
int iIdleTimeoutMs; // SoftTimer.h:128
```

`SoftTimer.c:784` 的 `st_timespec_add_ms(&node->deadline, &now, (long long)t_Alarm.iFirstDelayMs)`
是把 ms 直接叠加到 `timespec`，用户无法表达亚毫秒。
**最小可表达周期 = 1ms**（`iPeriodMs=0` 表示单次，不是 0ms 周期）。

### 1.2 内部时间表示：纳秒

`st_timespec_add_ms` 内部把 ms 拆成 `sec + nsec`，`timespec::tv_nsec` 是纳秒精度：

```c
long long add_nsec = (ms % 1000LL) * 1000000LL;   // SoftTimer.c:124
```

`pthread_cond_timedwait(CLOCK_MONOTONIC)` 也接受纳秒精度的绝对时刻。
**数据结构本身不是瓶颈**，如果哪天想扩到微秒，加一个 `iFirstDelayUs` 或用
`struct timespec` 入参即可。

### 1.3 实际精度：受内核与调度限制，IMX6ULL 上约 1–10 毫秒抖动

这才是"最小可用单位"的真答案，几个来源叠加：

| 来源 | 典型量级（IMX6ULL） | 说明 |
|---|---|---|
| 内核 tick (`CONFIG_HZ`) | 10ms (HZ=100) 或 4ms (HZ=250) | 若未开 `CONFIG_HIGH_RES_TIMERS`，`cond_timedwait` 醒来对齐 tick |
| hrtimer（若开启） | 亚毫秒 | 现代 arm-linux 一般开；`clock_getres(CLOCK_MONOTONIC)` 可查 |
| SCHED_OTHER 调度延迟 | 1–5ms | 调度线程被别的高优线程抢占时进一步放大 |
| SCHED_FIFO（`iSchedPriority>0`） | <1ms | Init 走 `SCHED_FIFO` 分支时抖动收敛 |
| 线程池投递（`ThreadPoolAddTaskTry`）| 0.1–1ms | 到点后从调度线程到 worker 的额外一跳 |
| `ST_POOL_ADDTASK_RETRY_MS` | 10ms | 池满退避（`SoftTimer_Main.h:50`）：池满场景下抖动地板抬到 10ms |

`需求文档.md §2.4` 直接标了 **"IMX6ULL 典型 1–10ms 抖动；毫秒级需求足够，微秒级请另寻方案"**，
这是设计定位。

### 1.4 一个易被忽视的下限：`ST_POOL_ADDTASK_RETRY_MS = 10`

一轮把 retry 从 1ms 抬到 10ms（M7），是为了避免"忙自旋 tick 抢锁"。
副作用是：**线程池饱和期间**，本轮触发被推迟至少 10ms。
所以在池压场景下，实际"最小周期"下限被抬到约 10ms，低于此值的周期定时器会持续 overrun 计数。

### 1.5 结论

- **规格上限**：1 ms（API 与 struct 都能表达）。
- **常态下限**：IMX6ULL 上 1–10 ms 抖动，`iSchedPriority>0` 时能压到低端。
- **饱和下限**：池满时被 `ST_POOL_ADDTASK_RETRY_MS=10ms` 保护性推迟。

**实践建议**：对于 IMX6ULL 上这套代码，把最短可靠周期定在
**10ms 左右**（FROM_END）或 **20ms 以上**（FROM_SCHEDULED，cb 需 <P）比较稳。
要更低（1–5ms）就得同时满足：`iSchedPriority>0`、CPU 空闲、`CONFIG_HIGH_RES_TIMERS=y`，
还得容忍偶发抖动。

## 二、`st_sched_thread` 优先级对定时任务的影响

**结论先说**：优先级只影响"到点的响应速度"，不影响"回调本身的执行速度"。
而且用高优先级时要小心一个隐藏的优先级反转。

### 2.1 会被优先级影响的三个阶段

调度线程在 `SoftTimer.c:826-910` 主循环里做三件事，每件事都被 `iSchedPriority` 直接决定：

#### (a) `cond_timedwait` 醒来后的调度延迟
`SoftTimer.c:851` 睡到 deadline，内核在 deadline 到达时把线程置为 runnable，
但**能否立刻上 CPU** 取决于优先级：

| iSchedPriority | 醒来 → 上 CPU 的典型延迟（IMX6ULL 单核） |
|---|---|
| 0（SCHED_OTHER，默认） | 1–5ms，被任何 CPU-bound 用户线程延后 |
| 1–49（SCHED_FIFO 低段） | <1ms，被更高优 FIFO 线程延后 |
| 50–99（SCHED_FIFO 高段） | <100µs，仅被 IRQ/softirq 抢占 |

`需求文档.md §2.4` 那句"1–10ms 抖动"就是 SCHED_OTHER 场景的量级。
开 FIFO 能把这一段压到亚毫秒。

#### (b) 投池路径的抢占窗口
`SoftTimer.c:863-866`：

```c
pthread_mutex_unlock(&mgr->mux);
int addret = ThreadAPI_ThreadPoolAddTaskTry(mgr->pool, st_wrapper_task, node);
pthread_mutex_lock(&mgr->mux);
```

这段无锁窗口内调度线程可被抢占。SCHED_OTHER 时如果这里被抢，`AddTaskTry`
返回时可能已经晚了几毫秒 → 直接体现在 `ulTotalFired` 增加时刻与理想 deadline 的偏差上。

#### (c) SetAlarm/Delete 的唤醒响应
`SoftTimer.c:811` 和 `SoftTimer.c:1122` 都会 `cond_signal(cv_sched)`。
signal 只是把 sched 线程标记为 runnable，**多久真正跑起来重排堆顶**依赖优先级。
低优先级时，一个"插队"的近 deadline 定时器可能要等好几毫秒才被 sched 观察到。

### 2.2 不会被 sched 优先级影响的阶段

#### (a) 用户回调的执行
回调跑在**线程池 worker** 上（`st_wrapper_task` 是投递到 `mgr->pool` 的任务），
worker 的优先级由 `ThreadManage` 的 `T_ThreadPoolConfig` 决定 —— 但当前
`SoftTimer.c:517-521` 只透传了 `iMinNum/iMaxNum/iQueueMaxSize/iIdleTimeoutMs`，
**没有透传 worker 优先级**。

意味着：**即使把 sched 拉到 FIFO 99，worker 仍是 SCHED_OTHER**，
回调开始执行的时刻仍受 worker 抢占影响。sched 帮你精准把任务扔进池子，
但从池子到 CPU 那一跳它管不了。

对 `ulTotalFired` 时间戳有用（这是"投递时刻"），
对"用户 cb 何时真正跑起来"效果有限。

#### (b) cb 本身的耗时
纯用户代码，与本库无关。

#### (c) 池满 requeue 的 10ms 退避
`ST_POOL_ADDTASK_RETRY_MS = 10` 是硬编码常量。
池满场景下延迟下限就是 10ms，sched 优先级再高也不能压下来。

### 2.3 一个真实存在的坑：`mgr->mux` 无优先级继承

这是被埋下的隐患。`SoftTimer.c:476` 初始化互斥锁：

```c
pthread_mutex_init(&pt_Mgr->mux, NULL);
```

`NULL` attr 意味着 mutex **没有优先级继承协议**（`PTHREAD_PRIO_NONE`）。

场景：
- 用户线程 T_user（SCHED_OTHER prio=0）在跑 `SetAlarm`，持有 `mgr->mux`；
- 与此同时另一个 CPU-bound 用户线程 T_bg（SCHED_OTHER prio=0）抢到 CPU，把 T_user 挤下；
- sched 线程（SCHED_FIFO prio=80）到点醒来，试图 `pthread_mutex_lock(&mgr->mux)`，阻塞；
- T_bg 继续跑 —— sched 高优等 T_user 释放锁，但 T_user 又被 T_bg 挤着 → **经典优先级反转**。

在单核 IMX6ULL 上更明显。虽然 mux 临界区都很短（SetAlarm 内几十行代码），
但只要用户业务里有长跑的 SCHED_OTHER 线程，这种情况就会出现。

**修法**（如果用户真的用高优 FIFO）：
```c
pthread_mutexattr_t mattr;
pthread_mutexattr_init(&mattr);
pthread_mutexattr_setprotocol(&mattr, PTHREAD_PRIO_INHERIT);
pthread_mutex_init(&pt_Mgr->mux, &mattr);
pthread_mutexattr_destroy(&mattr);
```

不是必须改的 —— V1.0 的定位是"毫秒级心跳"，用户很少上 FIFO。
但如果二轮之后有人真的用 `iSchedPriority=80` 去做 5ms 控制环，这就成必修项了。

### 2.4 overrun 计数与优先级的耦合

`SoftTimer.c:932-936` 的 FROM_END overrun：

```c
long long over_ms = st_timespec_diff_ms(cb_end, &node->last_scheduled);
if(over_ms > (long long)node->iPeriodMs) { ulTotalOverrun += 1; }
```

`cb_end - last_scheduled` 包含三段：
1. **sched 醒来 → AddTaskTry**（sched 优先级影响）
2. **AddTaskTry → worker 拿到任务开始跑**（worker 优先级影响）
3. **cb 本身耗时**（用户代码）

抬高 sched 优先级只压第 1 段。如果 worker 是 SCHED_OTHER，
第 2 段仍可能吃掉整个 period → **overrun 仍会计数**。
这就是为什么 `M11` 把 overrun 定义为"总回合时长"而不是"dispatch 延迟"——
反映端到端体感，而不是 sched 单点性能。

### 2.5 一句话总结

`iSchedPriority` **只是修调度线程的"到点判定"**，不是修回调的"执行优先级"。
如果需求是"cb 必须在 kT ± 1ms 触发并跑完"，光调 sched 不够，
还得让 `ThreadManage` 支持 worker 优先级透传（目前 V1.0 未做，属未来扩展点）。

### 2.6 可考虑的后续增强项

如果哪天要把这套用在硬实时场景，三处需要同步升级：

1. `T_SoftTimerConfig` 加 `int iPoolWorkerPriority`，透传到 `T_ThreadPoolConfig`（若 ThreadManage 支持）。
2. `mgr->mux` 加 `PTHREAD_PRIO_INHERIT`。
3. `ST_POOL_ADDTASK_RETRY_MS` 从常量改为配置项（10ms 对 5ms 周期是灾难）。

这些都不是 V1.0 缺陷，属于"用途升级时的必备扩展"，可以留档 V1.2 讨论。

## 三、能否用 Linux 内核级定时器替换当前方案

**结论**：可以借鉴，不建议替换。差异不在"能不能定时"，
而在**语义配套**——当前设计的核心价值不是"精准触发"，
是"任意时刻可安全删（含运行中删）+ 线程池并发执行 + 明确的所有权契约"。
这几件事内核定时器都不给你。

### 3.1 候选清单

Linux 提供了从老到新的几套：

| 机制 | 数量上限 | 触发方式 | Linux only? |
|---|---|---|---|
| `alarm()` / `SIGALRM` | 1 个/进程 | 信号 | POSIX |
| `setitimer(ITIMER_*)` | 3 个/进程（REAL/VIRT/PROF） | 信号 | POSIX |
| `timer_create()`（POSIX.1b） | N 个/进程 | 信号 或 helper 线程 | POSIX |
| `timerfd_create()` | N 个/进程 | 读 fd | Linux |
| `epoll_wait(timeout)` | 1 个"下次" | 主动 wait | Linux |

`alarm()`/`setitimer()` 直接排除——单一实例，无法承载 N 个并发定时器。
真正值得讨论的是后三个。

### 3.2 `timer_create()`（POSIX.1b）

看起来最像"替代品"：支持任意个、原生周期、走 hrtimer 精度。

**能做到**
- 每个 timer 是内核对象，无需自维护 min-heap；
- `timer_settime()` 一次设置 `it_value`（首次）+ `it_interval`（周期）；
- `timer_getoverrun()` 有原生 overrun 计数（"上次信号还未处理时又到点的次数"）；
- 支持 `CLOCK_MONOTONIC`。

**做不到（或成本很高）**

- **回调执行上下文**：只有两种。
  - `SIGEV_SIGNAL` → 信号处理器，只能调 async-signal-safe 函数（`malloc`/`printf`/`pthread_mutex_lock` 全部**禁用**）。用户的 `cb` 里想干点正常事？几乎不可能。
  - `SIGEV_THREAD` → glibc 实现是**每次 fire 派生一个临时线程**（或走一个 helper 线程分发），不受你控制。你无法限制并发度（线程池管不到），无法预分配栈，`fork bomb` 风险随任务频率线性上升。
- **D2 语义（他线程 Delete 正在运行的 cb 并等它结束）**：完全没有原语支持。`timer_delete()` 只保证不再触发下一次，不等已 fire 但还在跑的 cb。你得自己叠一层 `running_count + cond_wait`，等于把当前 `SoftTimer.c:1160-1188` 那段逻辑照抄一遍。
- **D1 语义（cb 内自删）**：`SIGEV_THREAD` 场景下，cb 所在线程 ≠ 一个稳定的 tid，`timer_delete()` 可以调，但"cb 结束时释放句柄"这套内存所有权还得自己写。
- **`FROM_END` 漂移模式**：内核只有"栅格式周期"（等价 FROM_SCHEDULED）。要 FROM_END 就得每次 cb 结束后 `timer_settime()` 重排为单次，等于把周期定时器降级成"一次性 + 手工续期"，反而更绕。
- **状态与统计**：`ulTotalFired`/`ulTotalCanceled`/`iActiveCount`/`iPeakActive` 这套账目仍要自己维护，内核不给。
- **优先级反转与 mutex**：同样存在，且信号路径引入更复杂的 sigmask 管理。
- **`sTimerName` 日志**：内核 timer 无名字，日志"哪个定时器 overrun 了"要自己维护映射。

**净收益评估**
换掉 min-heap + cond_timedwait 那 200 行代码，代价是砍掉线程池并发控制 +
Delete 语义 + FROM_END 模式 + 全部统计。**负收益。**

### 3.3 `timerfd`

比 `timer_create` 干净：把定时器变成一个 fd，`read()` 返回"这段时间 fire 了几次"。

**能做到**
- 用 `epoll_wait` 一次监听 N 个 timerfd，事件驱动；
- 完全避开信号；
- 天然多线程安全（fd 语义）；
- 精度同 hrtimer。

**做不到**
- 每个定时器占一个 fd → 进程 fd 上限限制（默认 1024，可调）。100 个定时器还行，1000 个开始要调 `ulimit`；
- 仍要**一个 dispatcher 线程 + 一个线程池**执行 cb（timerfd 本身不执行代码，只通知）；
- **Delete 时的 D2 语义**：一样得自己写 `running_count + cond_wait`；
- `FROM_END` 一样要手工续期；
- 统计一样得自己记；
- 相比当前实现，**唯一实质好处**是把"一个 cond_timedwait 睡到堆顶 deadline"换成
  "epoll_wait 醒来读 timerfd"。抖动可能小一点点（少一次堆顶重算），
  但 IMX6ULL 上淹没在 1–10ms 调度抖动里，测不出差别。

**净收益评估**：架构复杂度上升（要维护 fd 池 + epoll），换来微不足道的精度提升。**低收益。**

### 3.4 有意义的部分替换：`timerfd` 只替换 sched 线程的睡眠

如果非要用内核机制，唯一有意义的做法：

- 保留 min-heap、live_set、wrapper、Delete 五路径、线程池、统计——**不动**；
- 只把 `SoftTimer.c:851` 的
  `pthread_cond_timedwait(&mgr->cv_sched, &mgr->mux, &top->deadline)`
  换成 `epoll_wait` + 一个 timerfd（每次 `timerfd_settime` 重排到新的堆顶 deadline）；
- `cv_sched` 由 `signal(cv_sched)` 改为 `write(eventfd)`（或直接 rearm timerfd）。

**理论优点**
- 少一次条件变量 broadcast 的锁开销；
- 内核 hrtimer 精度略优（个位数百微秒）。

**实际代价**
- 增加 fd 管理（timerfd + eventfd = 2 个 fd，Init/Destroy 要多一层清理）；
- `pthread_cond_timedwait` 原子释放/重获 mutex 的语义丢了——要自己确保"发信号"与"睡下去"之间没竞态；
- 目标平台 IMX6ULL 的 hrtimer 精度本来就被 SCHED_OTHER 抖动掩盖，除非同时上 SCHED_FIFO + PREEMPT_RT，否则测不出差别。

**结论**：技术上可行，V1.0 不值得。

### 3.5 反过来问：哪些场景用内核定时器更合适？

给出一个"要不要用内核定时器"的判断表：

| 场景特征 | 推荐 |
|---|---|
| 全进程只有 1–3 个定时器，cb 极短（增计数、置 flag） | `setitimer` / `signalfd` 就够 |
| 已有 epoll 主循环（网络服务器），想加定时任务 | `timerfd` + 现有 epoll，天然融合 |
| 定时任务是"到点发一条消息"，无并发、无删除时序 | `timer_create(SIGEV_SIGNAL)` |
| 需要 N 个 cb 并行执行、支持取消、含运行中删、支持自删、要统计 | 当前 SoftTimer 设计 |
| 硬实时（周期误差 <100µs） | 内核定时器 + `SCHED_FIFO` + `PREEMPT_RT` 内核，SoftTimer 达不到 |
| 微秒级但非硬实时 | timerfd + SCHED_FIFO worker，SoftTimer 需扩 struct timespec 入参 |

### 3.6 为什么当前 pthread 方案是"合理默认"

回看 `需求文档.md §2.3` 那张架构图：**计时（调度线程）与执行（线程池）分离**。
这个设计选择的收益是：

- 计时层用 `cond_timedwait` —— 与所有 pthread 原语一致，无信号语义污染；
- 执行层用 `ThreadManage` 线程池 —— 并发度可控，与项目里 `WindowQueue`/`ThreadQueue` 的编程模型统一；
- Delete 的五路径能直接建立在 pthread 语义上（`pthread_equal`、`cond_wait`）。

换成内核定时器 = 计时层放到内核，但**执行层仍要自己写线程池**，
Delete 语义仍要自己写——最后收益是"计时层 200 行代码消失"，
代价是"执行层每种边界都要在信号/fd/线程三种上下文里重新推一遍"。

### 3.7 一句话总结

内核定时器**能定时，不能替代 SoftTimer**——
它给的是"到点通知"，SoftTimer 给的是"到点通知 + 并发执行 + 安全取消 + 明确所有权"这一整套。
前者是零件，后者是成品。要精度更高，直接把 sched 拉到 SCHED_FIFO + worker 也拉到 SCHED_FIFO 即可，
收益比换机制大得多。

**若一定要吸收内核机制的优点，务实做法是给未来 V1.1 留一个内部实现开关**
（`ST_CONFIG_USE_TIMERFD`），默认关，POC 时打开做对比测试。V1.0 就用当前方案，稳。

## 四、三个探讨的横向联系

三个话题各自独立，但落到"如何配置使用 SoftTimer"这一个决策点上，答案是耦合的：

| 用途 | 周期下限 | iSchedPriority 建议 | worker 优先级建议 | 备注 |
|---|---|---|---|---|
| 心跳/日志上报 | 500ms+ | 0 | 默认 | V1.0 目标场景，怎么配都行 |
| 传感器采样（软实时） | 20ms+ | 40–60（FIFO） | 需 ThreadManage 支持，V1.0 暂无 | 建议 FROM_END 避免风暴 |
| 控制环（软实时） | 10ms+ | 60–80（FIFO） | 需 ThreadManage 支持，V1.0 暂无 | FROM_SCHEDULED，cb 必须 <P |
| 硬实时 | — | — | — | 不适用本库，用 PREEMPT_RT + timerfd |

一个原则：**升配三件套要同步**——sched 优先级、worker 优先级、mutex PI 协议要一起上，
单点升级效果打折。V1.0 只提供第一件，另两件留到 V1.1/V1.2。
