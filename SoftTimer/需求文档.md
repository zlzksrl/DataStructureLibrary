> 本文档为 **SoftTimer 独立工具库**的需求规格（隶属 `DataStructureLibrary` 仓库）。
> 目标平台：IMX6ULL (ARM Linux) | 作者：zlzksrl | 草案日期：2026-07-13

## 一、原始需求（用户提供）

1. 启动定时器及配置函数。
2. `SoftTimerAPI_SetAlarm`。可以设置一个回调函数、回调函数参数、周期定时时间、首次定时启动时间。
3. 输入句柄和函数，删除定时。
4. 定时器可以使用一个线程池功能执行定时回调函数。
5. 要考虑同一个回调函数，可能注册成两个定时器的场景。
6. 要考虑在还没触发回调函数时，就删除定时器的场景。
7. 周期定时，用户自行删除定时。要考虑周期定时过程中另一线程删除定时的场景。
8. 在回调函数中删除定时器的场景。
9. 非周期定时（只启动一次的定时器），是否定时结束后自动删除？（可讨论）
10. 要考虑重复删除怎么报错。

## 二、需求理解与场景

### 2.1 应用场景
- 项目中多处需要"到点执行一段代码"的能力：传感器周期采样、心跳上报、超时看门狗、延迟一次性动作等。
- 需要用**软件方式**（用户态 pthread + `CLOCK_MONOTONIC`）实现，与 Linux 内核定时器解耦、可移植。
- 回调可能耗时（几十毫秒到数百毫秒），不能阻塞"计时"本身 —— 必须把**计时**与**执行**分离。

### 2.2 定位与已有库的关系
| 已有能力 | 局限 | 本库补充 |
|---------|------|---------|
| `ThreadManage` 线程池 | 只能"立刻投递任务"，不管时间 | 提供"到点自动投递"的调度层 |
| `ThreadQueue`         | 消息队列，无时间维度 | 用作调度线程→线程池的备选桥梁（本设计直接用 `ThreadPoolAddTask`） |
| Linux `timerfd` / `setitimer` | 与 pthread/线程池不易统一，信号模式复杂 | 用户态 pthread 定时，模型与本库其他模块一致 |

**结论**：新增独立库 **SoftTimer** —— 一个**支持单次/周期、句柄式管理、线程池执行回调、多线程安全**的软件定时器。

### 2.3 架构总览
```
             ┌────────────────────────────────────────────┐
用户线程 ────▶│ SetAlarm/Delete  → 调度器数据结构           │
             │      (按 deadline 排序 min-heap)            │
             │      唤醒信号: pthread_cond_signal          │
             └────────────────┬───────────────────────────┘
                              │  pthread_cond_timedwait(CLOCK_MONOTONIC)
                              │  睡到最早 deadline 或被 signal 唤醒
                              ▼
                    ┌─────────────────────┐
                    │  调度线程 (1 个)     │  ── 到点摘出 timer
                    └──────────┬──────────┘
                               │ ThreadPoolAddTask(cb, ctx)
                               ▼
                    ┌─────────────────────┐
                    │  线程池 (N 个 worker) │  ── 实际执行 cb
                    └─────────────────────┘
```
- **调度线程**：库启动时创建 1 个，仅负责"睡到下一个 deadline → 摘取到期定时器 → 投递给线程池"，本身不执行用户回调。
- **线程池**：库启动时创建（复用 `ThreadAPI_ThreadPoolCreate`），并发执行到期回调。
- **周期定时器**：投递到线程池后，调度线程立即计算下一个 deadline 重新入堆。

### 2.4 计时机制（非轮询，事件驱动 + 精确睡眠）
- **时钟源**：`clock_gettime(CLOCK_MONOTONIC, &ts)`（内核单调时钟，不受 `date`/NTP 影响）。
- **睡眠原语**：`pthread_cond_timedwait`，其 cond var 在初始化时通过 `pthread_condattr_setclock(&attr, CLOCK_MONOTONIC)` **绑定 CLOCK_MONOTONIC**（否则 `timedwait` 默认吃 `CLOCK_REALTIME`，改系统时间会导致定时器错乱）。
- **调度线程主循环**（伪码）：
  ```c
  while (!stop) {
      lock(mgr);
      if (heap_empty()) {
          pthread_cond_wait(&cv, &mux);            // 无定时器：无限睡
      } else {
          struct timespec dl = heap_top()->deadline;   // 绝对时刻
          pthread_cond_timedwait(&cv, &mux, &dl);      // 精准睡到那一刻
      }
      // 醒来两种可能：
      //   (1) 到点：摘出所有 deadline<=now 的定时器，投递线程池；
      //       周期定时器重算 deadline 后重新入堆
      //   (2) SetAlarm/Delete 用 cond_signal 唤醒：重新看堆顶，睡到新 deadline
      process_due(mgr);
      unlock(mgr);
  }
  ```
- **不消耗 CPU**：整个线程在 `cond_timedwait` 里由内核挂起，无轮询。
- **动态响应**：`SetAlarm` 若插入的 deadline 早于当前堆顶，即刻 `cond_signal` 唤醒调度线程重排；`Delete` 删堆顶同理。
- **不采用轮询**（例如 `while(1) { sleep(1ms); check(); }`）：CPU 常驻消耗、精度更差、无法响应新注册的更早 deadline。
- **精度**：受内核 tick 与调度延迟影响，IMX6ULL 典型 1–10ms 抖动；毫秒级需求足够，微秒级请另寻方案。

## 三、功能规格

### 3.1 生命周期
- `SoftTimerAPI_Init(&mgr, t_Config)`：分配管理器 `T_SoftTimerMgr`，创建内部调度线程与线程池（池参数见 3.2）。`t_Config` **按值传递**（拷贝入库），调用后原变量即可释放/复用。
- `SoftTimerAPI_Destroy(&mgr)`：停止调度线程、销毁线程池、清理所有未触发定时器（不再回调）、`*mgr` 置 NULL。
- 单进程一般只需一个 `T_SoftTimerMgr` 实例，允许多实例（各自独立线程池与调度线程）。

### 3.2 初始化配置 `T_SoftTimerConfig`
- `char sMgrName[32]`：管理器名称（内联定长数组，最多 31 字符 + '\0'），用于日志与线程命名。
- `int iMinNum / iMaxNum / iQueueMaxSize / iIdleTimeoutMs`：透传给内部 `T_ThreadPoolConfig`。
- `int iSchedPriority`：调度线程优先级（0=默认，>0=SCHED_FIFO/RR）。

**为何用内联定长数组**：结构体按值传递时字符串**随结构体一起拷贝**，无需库内 `strdup`、无生命周期烦恼、无额外堆分配。C99 支持 `.sMgrName = "app"` 指定初始化。

### 3.3 注册定时器 `SoftTimerAPI_SetAlarm`
```c
int SoftTimerAPI_SetAlarm(T_SoftTimerMgr *pt_Mgr,
                          const T_SoftTimerAlarm t_Alarm,
                          T_SoftTimerHandle **ppt_Handle);
```
- `T_SoftTimerAlarm` 字段（**按值传递**）：
  - `SoftTimerCb cb`：回调函数（不能为 NULL）。
  - `void *user_ctx`：回调上下文，透传。
  - `int iFirstDelayMs`：**首次**触发相对于 `SetAlarm` 调用时刻的延迟（≥0，0 表示尽快触发）。
  - `int iPeriodMs`：**周期**（>0=周期定时器；0=单次定时器）。
  - `E_SoftTimerPeriodMode ePeriodMode`：周期漂移模式（详见 4.1）。**枚举 0 = 默认 `FROM_END`（间隔精确，无风暴）**；`FROM_SCHEDULED`（定点精确，需用户保证 `period > 回调耗时`）。单次定时器该字段无意义。
  - `char sTimerName[32]`：定时器名（内联定长数组，最多 31 字符 + '\0'），用于日志。空串表示无名字。
- 内部：
  1. 读 `CLOCK_MONOTONIC`，计算 `deadline = now + iFirstDelayMs`；
  2. 分配 `T_SoftTimer` **内部节点**入 min-heap；
  3. **分配 `T_SoftTimerHandle` 句柄结构体**（内部含 `ullId` 单调递增序号，仅用于日志），登记到内部**live_set**（按指针地址索引）；
  4. `pthread_cond_signal` 唤醒调度线程重算睡眠。
- **入参约束**（对齐 `Init` 风格）：
  - `ppt_Handle != NULL`；
  - `*ppt_Handle == NULL`（防止覆盖已有句柄导致内存泄漏）。
  - 成功后 `*ppt_Handle` 指向库分配的句柄结构体。
- **同一 `cb`+`user_ctx` 可重复调用**，每次得到**不同的指针**（对应第 5 点）。
- 返回 `0` 成功；`-1` 参数无效（含 `ppt_Handle==NULL` 或 `*ppt_Handle!=NULL`）；`-2` 管理器已销毁/停止。

### 3.4 回调原型 `SoftTimerCb`
```c
typedef void (*SoftTimerCb)(T_SoftTimerHandle *pt_Handle, void *user_ctx);
```
- 传入 `pt_Handle` 便于回调内自删除或续订，不需要用户自己在 `user_ctx` 里塞一份。
- ⚠ 回调内如需自删除，用户须**自行准备一个二级指针变量**（例如 `T_SoftTimerHandle *self = pt_Handle; SoftTimerAPI_Delete(mgr, &self);`）—— `pt_Handle` 直接传给用户是**只读身份**，不是可 null 化的用户变量。

### 3.5 删除定时器 `SoftTimerAPI_Delete`
```c
int SoftTimerAPI_Delete(T_SoftTimerMgr *pt_Mgr,
                        T_SoftTimerHandle **ppt_Handle);
```
- **入参约束**（对齐 `Destroy` 风格）：
  - `ppt_Handle != NULL`；
  - `*ppt_Handle != NULL`。
  - 成功后 `*ppt_Handle` 被置为 NULL。
- **身份判定**：仅按 `*ppt_Handle` 的**指针地址**在 `live_set` 中查询成员身份，**不解引用用户指针**，野指针也不会 crash。
- 语义（对应原始需求 6/7/8/10）：
  - **未触发时删除**：从堆中摘除内部节点、释放节点、从 live_set 移除句柄、释放句柄结构体、`*ppt_Handle = NULL`。返回 `0`。
  - **正在线程池执行时删除**（另一线程调用）：标记 `is_canceled=1`，**阻塞等待**当前回调执行完毕（`pthread_cond_wait` 等 `running_count==0`）后回收内部节点、回收句柄、置 NULL。返回 `0`。
    - 保证：`Delete` 返回后，回调**一定不会再被调用**（哪怕是周期的下一轮），且**此刻**没有该定时器的回调在执行。
  - **回调内自删除**（调用者线程 == 当前正在执行的 worker）：检测 `pthread_equal(pthread_self(), running_tid)`，跳过 wait 直接标记，**不阻塞**（否则死锁）；内部节点由回调返回后 wrapper 释放；**句柄结构体在 wrapper 里回收**。用户变量在 Delete 返回时置 NULL。返回 `0`。
  - **单次定时器已 fire 完（墓碑）**：Delete 视为"回收墓碑"，释放句柄结构体、置 NULL。返回 `0`。
  - **重复删除**：第二次调用时 `*ppt_Handle==NULL` → 参数无效 → 返回 `-1`。**天然规避**双重释放。
  - **陌生指针 / 别的 mgr 的 handle / 已被 Destroy 扫掉的野指针**：`*ppt_Handle != NULL` 但不在 `live_set` → 返回 `-3`。
- ⚠ **契约**：句柄指针**不要 alias 到多个变量**（避免 ABA 风险）；对同一变量调用 Delete 一次即可，第二次调用会因 `*ppt_Handle==NULL` 自然返回 `-1`。

> 用户原始需求第 3 点"输入句柄和函数，删除定时"：由于第 5 点允许同一 `cb` 注册多次，仅凭 `cb` 无法唯一识别 → **主删除 API 用 handle 指针**。若确需"按 cb 批量删除"，见 3.7 的可选扩展。

### 3.6 单次定时器的收尾（原始需求第 9 点）
- **内部节点**：单次定时器的回调执行完毕后**自动释放**（用户不必操心节点内存）。
- **句柄结构体**：**仍归用户所有**，须由 `SoftTimerAPI_Delete(&handle)` 或 `SoftTimerAPI_Destroy(&mgr)` 回收（对齐 `Init/Destroy` 强制配对约定）。
- 三条实际用法路径：
  1. **fire 前 Delete**：正常取消。返回 `0`，句柄回收，用户变量置 NULL。
  2. **fire 后 Delete**：墓碑回收，返回 `0`，用户变量置 NULL。
  3. **fire 后不 Delete**（fire-and-forget）：句柄结构体作为墓碑滞留在 live_set，等 `Destroy` 扫尾清理；用户变量将保留一个"墓碑指针"，用户须自觉不再使用。
- 决策见 D5。

## 四、并发场景与线程安全承诺

针对原始需求 5/6/7/8，逐条给出保证：

| 场景 | 需求点 | 保证 |
|------|--------|------|
| 同一 `cb` 注册两个定时器 | 5 | 每次 SetAlarm 返回不同的 handle 指针，节点独立；两者可分别删除、独立触发。 |
| 未触发前删除 | 6 | 从堆中摘除内部节点、释放节点、回收句柄结构体、`*ppt_Handle` 置 NULL；`Delete` 返回 `0`，`cb` 保证不会被调用。 |
| A 线程 SetAlarm，B 线程 Delete（回调正在执行） | 7 | B 阻塞至 cb 结束再返回；返回后 cb 保证不再触发；节点与句柄均已回收，`*ppt_Handle` 置 NULL。 |
| 回调内 `Delete(自身)` | 8 | 检测同线程，不阻塞；标记取消 + 释放句柄 + 用户变量置 NULL；内部节点在 wrapper 收尾时释放；对周期定时器等效于"这是最后一次"。 |
| 重复删除（同变量二次 Delete） | 10 | 第二次时 `*ppt_Handle==NULL` → 参数无效，返回 `-1`。天然规避双重释放。 |
| 陌生 / 别的 mgr / 已被 Destroy 扫掉的野指针 | 10 | `*ppt_Handle != NULL` 但不在 live_set → 返回 `-3`。 |
| 回调内 `SetAlarm` 新定时器 | (隐含) | 允许，与外部 SetAlarm 等价（不会死锁）。 |

**关键实现要点**：
- **句柄身份 = 指针地址**（非 ullId）：内部维护 `live_set`（按指针地址索引），Delete 时**只做集合成员查询、不解引用用户指针**，野指针也不会 crash。
- **`ullId`（单调递增 uint64）保留在句柄结构体内**，但仅用于日志/stats（如 `"timer #42 canceled"`），不参与身份判定；未来可自由扩展字段而不影响 ABI。
- **ABA 契约**：句柄指针**不要 alias 到多个变量**。若用户把 `T_SoftTimerHandle *` 复制到多处并在 Delete 之后仍访问陈旧副本，属于 use-after-free 未定义行为（与 `fclose(fp)` 后使用 `fp` 同性质）。
- `T_SoftTimer` 内部节点带 `int running_count`、`int is_canceled`、`pthread_t running_tid`、`T_SoftTimerHandle *handle`（回指句柄用于 wrapper 释放）。
- 线程池 wrapper 逻辑（伪码）：
  ```
  wrapper(node):
      lock(mgr)
      if node->is_canceled: unlock; goto RELEASE
      node->running_tid   = pthread_self()
      node->running_count = 1
      unlock
  
      user_cb(node->handle, node->user_ctx)   // 锁外调用
  
      lock(mgr)
      node->running_count = 0
      if node->is_canceled or node->period == 0:
          goto RELEASE (in-lock, 单次定时器 fire 后到此)
      else:
          node->deadline = compute_next(node->period_mode, ...)   // 见 4.1
          heap_push(node); cond_signal(sched)
      unlock; return
  
  RELEASE:
      free(node)                                  // 内部节点释放
      if node->period == 0 and !was_canceled:
          // 单次 fire 完，句柄变墓碑（is_consumed=1），留待用户 Delete 或 Destroy 扫尾
          node->handle->is_consumed = 1
      else:
          // Delete 路径：句柄由 Delete 调用者负责释放（Delete 返回前置 NULL）
          // 若是"回调内自删除"路径，则由 wrapper 释放句柄（Delete 已提前返回）
          maybe_free_handle(...)
      cond_broadcast(delete_done)                 // 唤醒等待中的外部 Delete
      unlock
  ```

### 4.1 周期定时器的漂移模式（`ePeriodMode`，每个定时器独立配置）

用户在 `SetAlarm` 时选择模式；两种模式各自有明确适用场景，**mgr 层不设默认策略、由调用方按任务性质决定**。

#### 模式 A：`SOFTTIMER_PERIOD_FROM_END = 0`（**默认**，间隔精确）
- 规则：`next_deadline = 回调结束时刻 + iPeriodMs`。
- 语义：两次回调之间的**空闲间隔**恒为 `period`。
- 举例：`period=100ms`，某次回调耗时 150ms → 实际间隔 100ms（cb 结束后再等 100ms），单次周期从 cb 开始到下次 cb 开始 = 250ms。
- **保证**：**永远不会补发**、**永远不会挤压线程池**。回调慢了，整体节奏被拖慢，仅此而已。
- **overrun 判定**：`cb_end - last_scheduled > iPeriodMs` 时 `ulTotalOverrun += 1`。语义是"总回合时长（dispatch 延迟 + cb 耗时）超过一个周期"。若纯粹的 dispatch 排队延迟很大但 cb 秒回，且相加后仍不超周期，不计 overrun。
- 适用：心跳、看门狗喂狗、日志上报、"至少每隔 X 秒做一次" 类任务。

#### 模式 B：`SOFTTIMER_PERIOD_FROM_SCHEDULED = 1`（定点精确）
- 规则：`next_deadline = prev_scheduled + iPeriodMs`。
- 语义：回调按 `t=0, T, 2T, 3T…` 的**时间栅格**触发，累计不漂移。
- **前提**：用户须自行保证 `period > 回调耗时`（此模式设计假设）。
- **overrun 保护**（关键，防止线程池被打爆）：
  - 若 `next_deadline <= now`（说明这次回调超时了），**不做无脑 catch-up**（那会连续投递 N 个任务），而是 **snap 到下一个未来 grid 点**：
    ```
    missed  = ceil((now - prev_scheduled) / period)   // 至少 1
    next    = prev_scheduled + missed * period        // 下一个未来 grid
    overrun_count += (missed - 1)                     // 记录跳过的 grid 数
    ```
  - 效果：偶发超时只会**丢帧**，不会连投；栅格对齐得以维持。
- 适用：控制环、采样时基、"必须在时间点 kT 触发"类任务。
- ⚠ 与模式 A 的行为差异总结：
  | 场景 | FROM_END | FROM_SCHEDULED |
  |------|----------|----------------|
  | 回调不超时 | grid 对齐（等效） | grid 对齐（等效） |
  | 回调偶尔超时 | 后续节奏永久滞后 | 丢当前 grid、后续对齐 |
  | 回调持续超时 | 单调拖慢，无风暴 | 持续丢帧+overrun 计数，无风暴 |

#### `ulTotalOverrun` 语义（三合一）
`ulTotalOverrun` 是一个粗粒度的"背压/延迟"指示，来源合并计数：
- **模式 A（FROM_END）**：每次 **cb 完成时刻晚于"本次预定 deadline + iPeriodMs"** 时 +1。
  即以 `cb_end` 与 `last_scheduled` 差值 > period 作为判定基准，衡量"总回合时长"（dispatch 延迟 + cb 耗时）是否超过一个周期。
- **模式 B（FROM_SCHEDULED）**：每次 snap 时 `+= (missed - 1)`（跳过的 grid 数）；栅格恰好落在 `cb_end` 时不计。
- **池满 requeue（模式无关）**：调度线程投池失败推迟本次 deadline 时 +1。

三种来源共用一个计数字段，用于粗粒度背压监控；如需区分具体成因，需依赖日志。通过 `StatsGet` 上报，用于判断是否需要调大 period 或加线程池 worker。

决策见 D4。

### 4.2 时间源
- 统一使用 **`CLOCK_MONOTONIC`**（`clock_gettime` + `pthread_condattr_setclock`）：不受系统时间调整影响，适合"相对延迟"语义。
- 详见 2.4 计时机制。

### 4.3 管理器可重复创建
- `T_SoftTimerMgr` 支持 `Init → Destroy → Init` 循环创建（顺序调用，无残留）。
- 每次 `Init` 得到独立实例（新调度线程、新线程池、handle 序号从 1 重新计数），互不干扰。
- 单进程内也允许**多个 `T_SoftTimerMgr` 实例并存**（各自持有线程池与调度线程）。

## 五、内存管理
- `T_SoftTimer` 节点、`T_SoftTimerHandle` 句柄结构体均由库内部 `malloc/free`，用户无需管理。
- `T_SoftTimerConfig` / `T_SoftTimerAlarm` **按值传递**：库在 Init/SetAlarm 内**立即拷贝**所需字段（含内联 `sMgrName[32]` / `sTimerName[32]`）；调用返回后用户可立即释放/复用原变量。
- `user_ctx` 由用户所有：库只透传指针，不负责生命周期。
  - **约束**：用户须保证 `user_ctx` 在**从 SetAlarm 成功返回到 Delete 返回（或单次定时器 cb 结束）之间**有效。
- 无 `release_callback`（因为节点内没有代管的用户堆内存）。

## 六、错误码约定（与库内其他模块保持一致）
| 错误码 | 宏 | 含义 |
|--------|-----|------|
| `0`  | `SOFTTIMER_OK`         | 成功 |
| `-1` | `SOFTTIMER_ERR_PARAM`  | 参数无效或未初始化（含 `ppt_Handle==NULL` / SetAlarm 时 `*ppt_Handle!=NULL` / Delete 时 `*ppt_Handle==NULL`——**重复删除自然进这里**） |
| `-2` | `SOFTTIMER_ERR_CLOSED` | 管理器已停止/销毁 |
| `-3` | `SOFTTIMER_ERR_NOT_FOUND` | Delete：`*ppt_Handle` 非 NULL 但不在 `live_set`（陌生指针 / 别的 mgr 的 handle / 野指针） |

## 七、关键设计决策（已确认 ✓）

| 编号 | 决策 | 选定 |
|------|------|------|
| **D1** | 计时后端 | pthread + `pthread_cond_timedwait(CLOCK_MONOTONIC)` + 1 个调度线程 ✓ |
| **D2** | 到期结构 | **内建 min-heap**（按 deadline 排序）；不依赖外部 RB-tree ✓ |
| **D3** | 回调执行 | 复用 `ThreadManage` 线程池；SoftTimer **内部**创建并持有 ✓ |
| **D4** | 周期漂移策略 | **每个定时器独立配置** `ePeriodMode`：`FROM_END`（默认，间隔精确、无风暴）/ `FROM_SCHEDULED`（定点精确，overrun 时 snap 到下一未来 grid，不 catch-up 不风暴） ✓ |
| **D5** | 单次定时器 fire 后 | **内部节点自动释放**；**句柄结构体仍归用户所有**，靠 `Delete(&h)` 或 `Destroy(&mgr)` 回收（对齐 Init/Destroy 强制配对） ✓ |
| **D6** | V1.0 API 范围 | 仅 5 个：`Init/Destroy/SetAlarm/Delete/StatsGet`；`Adjust/Pause/DeleteByCallback` 留 V1.1 ✓ |
| **D7** | 句柄形式与身份 | **不透明结构体指针** `T_SoftTimerHandle *`（前向声明），二级指针风格 API 对齐 `Init/Destroy`；身份判定用**指针地址**（`live_set` 成员查询，不解引用用户指针）；结构体内保留 `ullId` 仅供日志 ✓ |
| **D8** | 回调签名 | `void cb(T_SoftTimerHandle *pt_Handle, void *user_ctx)`（含只读身份 handle 便于日志/续订；回调内自删须自行准备二级指针变量） ✓ |
| **D9** | 线程池所有权 | **内部**持有：`Init` 创建、`Destroy` 销毁；不接受外部注入 ✓ |
| **D10** | 命名前缀 | `SoftTimerAPI_`，句柄类型 `T_SoftTimerMgr` / `T_SoftTimerHandle` ✓ |
| **D11** | 绝对时间接口 | **不做**（本项目场景仅需相对延迟） ✓ |
| **D12** | 时钟源 | `CLOCK_MONOTONIC`；cond var 通过 `pthread_condattr_setclock` 绑定 ✓ |
| **D13** | 管理器可重复创建 | 支持 `Init→Destroy→Init` 循环；允许多实例并存 ✓ |

## 九、公开 API（定稿，V1.0）

```c
/* ---- 前向声明与类型 ---- */
typedef struct T_SOFTTIMERMGR    T_SoftTimerMgr;      /* 管理器句柄，内部定义 */
typedef struct T_SOFTTIMERHANDLE T_SoftTimerHandle;   /* 定时器句柄，内部定义；用户以指针形式持有 */

/* 回调原型：pt_Handle 仅作只读身份（日志/续订用），如需自删须自行准备二级指针变量 */
typedef void (*SoftTimerCb)(T_SoftTimerHandle *pt_Handle, void *user_ctx);

/* 周期漂移模式（详见 4.1） */
typedef enum
{
    SOFTTIMER_PERIOD_FROM_END       = 0,  /**< 默认：next = cb结束时 + period；间隔精确、无风暴。心跳/上报类。 */
    SOFTTIMER_PERIOD_FROM_SCHEDULED = 1,  /**< 定点精确：next = 上次预定 + period；overrun 时 snap 到下一未来 grid。控制环/采样类，需保证 period > cb 耗时。 */
} E_SoftTimerPeriodMode;

/* Alarm 配置 */
typedef struct T_SOFTTIMERALARM
{
    SoftTimerCb           cb;              /**< 回调函数，不能为 NULL */
    void                 *user_ctx;        /**< 回调上下文，可为 NULL */
    int                   iFirstDelayMs;   /**< 首次延迟(ms)，>=0 */
    int                   iPeriodMs;       /**< 周期(ms)，>0=周期定时，0=单次定时 */
    E_SoftTimerPeriodMode ePeriodMode;     /**< 周期漂移模式，0=FROM_END（默认）；单次定时器忽略 */
    char                  sTimerName[32];  /**< 定时器名（内联定长，最多 31 字符+'\0'），日志用；空串表示无名 */
} T_SoftTimerAlarm;

/* 管理器配置 */
typedef struct T_SOFTTIMERCONFIG
{
    char sMgrName[32];            /**< 管理器名（内联定长，最多 31 字符+'\0'） */
    int  iMinNum;                 /**< 线程池最小线程数 */
    int  iMaxNum;                 /**< 线程池最大线程数 */
    int  iQueueMaxSize;           /**< 线程池任务队列最大长度 */
    int  iIdleTimeoutMs;          /**< 线程池空闲缩容超时(ms)，0=禁用 */
    int  iSchedPriority;          /**< 调度线程优先级，0=默认 */
} T_SoftTimerConfig;

/* 运行统计 */
typedef struct T_SOFTTIMERSTATS
{
    unsigned long ulTotalSet;
    unsigned long ulTotalFired;
    unsigned long ulTotalCanceled;
    unsigned long ulTotalOverrun;
    int           iActiveCount;
    int           iPeakActive;
} T_SoftTimerStats;

/* ---- 生命周期 ---- */
/**
 * 初始化：调用前 ppt_Mgr 非 NULL 且 *ppt_Mgr 必须为 NULL；成功后 *ppt_Mgr 指向新分配的管理器。
 * t_Config 按值传递（含 sMgrName 内联数组），调用后原变量可释放/复用。
 * @return 0 成功；-1 参数无效（含 *ppt_Mgr 非 NULL）；-2 内部资源创建失败（线程池/线程）
 */
int SoftTimerAPI_Init   (T_SoftTimerMgr **ppt_Mgr, const T_SoftTimerConfig t_Config);

/**
 * 销毁：调用前 ppt_Mgr 非 NULL 且 *ppt_Mgr 必须非 NULL；成功后 *ppt_Mgr 置 NULL。
 * 会先停止调度线程、销毁线程池，然后扫尾所有残留句柄（含单次墓碑）。
 * @return 0 成功；-1 参数无效（含 *ppt_Mgr==NULL）
 */
int SoftTimerAPI_Destroy(T_SoftTimerMgr **ppt_Mgr);

/* ---- 注册与删除 ---- */
/**
 * 注册定时器：调用前 ppt_Handle 非 NULL 且 *ppt_Handle 必须为 NULL；
 * 成功后 *ppt_Handle 指向新分配的句柄结构体。
 * t_Alarm 按值传递（含 sTimerName 内联数组），调用后原变量可释放/复用。
 * @return 0 成功；-1 参数无效（含 *ppt_Handle 非 NULL）；-2 管理器已销毁
 */
int SoftTimerAPI_SetAlarm(T_SoftTimerMgr *pt_Mgr,
                          const T_SoftTimerAlarm t_Alarm,
                          T_SoftTimerHandle **ppt_Handle);

/**
 * 删除定时器：调用前 ppt_Handle 非 NULL 且 *ppt_Handle 必须非 NULL；
 * 成功后释放句柄结构体并将 *ppt_Handle 置 NULL；对同一变量的第二次调用
 * 因 *ppt_Handle==NULL 自然返回 -1（重复删除天然规避）。
 * @return 0 成功；-1 参数无效（含 *ppt_Handle==NULL，即重复删除）；
 *         -2 管理器已销毁；-3 *ppt_Handle 非 NULL 但不在 live_set
 */
int SoftTimerAPI_Delete  (T_SoftTimerMgr *pt_Mgr, T_SoftTimerHandle **ppt_Handle);

/* ---- 运行统计 ---- */
int SoftTimerAPI_StatsGet(T_SoftTimerMgr *pt_Mgr, T_SoftTimerStats *pt_Stats);
```

**典型调用示例**（C99 指定初始化器）：
```c
/* 1. 初始化管理器 */
T_SoftTimerMgr *mgr = NULL;
T_SoftTimerConfig cfg = {
    .sMgrName       = "app_timer",   /* 内联数组，字符串字面量直接赋值 */
    .iMinNum        = 2,
    .iMaxNum        = 8,
    .iQueueMaxSize  = 64,
    .iIdleTimeoutMs = 5000,
    .iSchedPriority = 0,
};
if (SoftTimerAPI_Init(&mgr, cfg) != 0) { /* 失败处理 */ }

/* 2. 注册周期定时器（1 秒心跳，FROM_END 模式） */
T_SoftTimerHandle *h_heartbeat = NULL;
T_SoftTimerAlarm alarm = {
    .cb             = on_heartbeat,
    .user_ctx       = &ctx,
    .iFirstDelayMs  = 1000,
    .iPeriodMs      = 1000,
    .ePeriodMode    = SOFTTIMER_PERIOD_FROM_END,
    .sTimerName     = "heartbeat",
};
if (SoftTimerAPI_SetAlarm(mgr, alarm, &h_heartbeat) != 0) { /* 失败处理 */ }

/* 3. 删除（h_heartbeat 成功后被置 NULL） */
SoftTimerAPI_Delete(mgr, &h_heartbeat);

/* 4. 销毁（mgr 成功后被置 NULL） */
SoftTimerAPI_Destroy(&mgr);
```
